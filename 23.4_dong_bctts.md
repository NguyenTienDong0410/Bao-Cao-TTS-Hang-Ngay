### 1. Quản lý User và Group

Trước khi tiến hành phân quyền, quản trị viên cần nắm rõ cách quản lý các đối tượng được phân quyền. Linux lưu trữ thông tin người dùng trong các file hệ thống và quản lý thông qua các lệnh chuyên biệt.

#### 1.1. Các file cấu hình cốt lõi

**`/etc/passwd`**: Chứa thông tin cơ bản của tất cả user trên hệ thống, bao gồm tên đăng nhập, UID, GID, thư mục home và shell mặc định. File này có thể đọc bởi tất cả người dùng vì không chứa thông tin nhạy cảm.

<img width="1158" height="513" alt="image" src="https://github.com/user-attachments/assets/1627ebe9-68a0-4e19-93e3-e7a3094db8db" />

**`/etc/shadow`**: Chứa mật khẩu đã được mã hóa (hash) và các chính sách hết hạn mật khẩu. Vì lý do bảo mật, chỉ user `root` mới có quyền đọc file này.

<img width="828" height="630" alt="image" src="https://github.com/user-attachments/assets/d724c1ca-55c8-43d1-b1f2-c9522a0afdd7" />

**`/etc/group`**: Chứa thông tin về các nhóm trên hệ thống, bao gồm tên nhóm (Group name), mã số nhóm (GID) và danh sách các thành viên trong từng nhóm.

<img width="542" height="480" alt="image" src="https://github.com/user-attachments/assets/0f44eb9a-29d5-4c87-92eb-ba319dff917c" />

#### 1.2. Các lệnh quản lý cơ bản

**Tạo mới:**

- `useradd -m alice`: Tạo user mới tên là `alice`. Tham số `-m` để hệ thống tự động tạo thư mục `/home/alice`.
- `groupadd devteam`: Tạo một group mới tên là `devteam`.

**Thiết lập mật khẩu:**

- `passwd alice`: Đặt hoặc thay đổi mật khẩu cho user `alice`.

**Chỉnh sửa và thêm vào nhóm:**

- `usermod -aG devteam alice`: Thêm user `alice` vào group `devteam`. Tham số `-aG` có nghĩa là "append group" — thêm vào nhóm mới mà không xóa khỏi các nhóm hiện tại.

<img width="617" height="231" alt="image" src="https://github.com/user-attachments/assets/72fd7720-c63d-4bc4-83d1-2346b44d0285" />

**Xóa:**

- `userdel -r alice`: Xóa user `alice` và toàn bộ dữ liệu trong thư mục home của họ.
- `groupdel devteam`: Xóa group `devteam`.

### 2. Phân quyền đặc biệt: Sticky Bit

Trong quản trị hệ thống Linux, đặc biệt khi quản lý các máy chủ chia sẻ tài nguyên hoặc thư mục chứa file tạm (như `/tmp`), quản trị viên thường gặp bài toán: *Làm sao để cho phép nhiều người dùng cùng tạo và chỉnh sửa file trong một thư mục, nhưng không ai được phép xóa file của người khác?* Cơ chế **Sticky Bit** ra đời để giải quyết vấn đề này.

#### 2.1. Khái niệm và đặc điểm

**Định nghĩa:** Sticky Bit là một quyền đặc biệt (special permission bit) được áp dụng trên thư mục, không phải trên file.

**Tác dụng:** Khi một thư mục được bật Sticky Bit, dù thư mục đó có cấp quyền ghi (`w`) đầy đủ cho tất cả mọi người (777), thì **chỉ có chủ sở hữu của file, chủ sở hữu của thư mục, hoặc user `root` mới có quyền xóa hoặc đổi tên file đó**. Đây là biện pháp bảo vệ dữ liệu dùng chung hiệu quả.

**Nhận diện:** Khi kiểm tra bằng lệnh `ls -l`, ký tự cuối cùng trong chuỗi phân quyền sẽ là chữ **`t`** (thay vì `x`). Ví dụ: `drwxrwxrwt`.

**Cách thiết lập:** Dùng lệnh `chmod +t <thư_mục>` hoặc thêm số `1` vào trước bộ 3 số phân quyền, ví dụ: `chmod 1777`.

#### 2.2. Thực hành minh họa Sticky Bit

Mục tiêu: Tạo một thư mục dùng chung cho nhiều user, bật Sticky Bit và chứng minh rằng user này không thể xóa file của user kia.

**Bước 1: Tạo thư mục chia sẻ và cấu hình Sticky Bit**

```bash
# 1. Tạo một thư mục chia sẻ dữ liệu
sudo mkdir /tmp/shared_data

# 2. Cấp toàn quyền (777) và bật Sticky Bit (thêm số 1 ở đầu)
sudo chmod 1777 /tmp/shared_data

# 3. Kiểm tra phân quyền — kết quả mong đợi: drwxrwxrwt
ls -ld /tmp/shared_data
```

<img width="764" height="150" alt="image" src="https://github.com/user-attachments/assets/d3ac2b65-e25c-4c19-9a49-57319aff6fee" />

**Bước 2: User Alice tạo file vào thư mục chung**

```bash

#    (Nếu chưa có user này, chạy: sudo useradd -m alice)
sudo su - alice

# 2. Alice tạo một file vào thư mục chung
touch /tmp/shared_data/alice_report.txt

# 3. Kiểm tra file vừa tạo — kết quả mong đợi: file thuộc sở hữu của alice

```

```bash

```

**Bước 3: User Bob cố tình xóa file của Alice**

```bash

#    (Nếu chưa có user này, chạy: sudo useradd -m bob)
```

### 3. Lệnh thay đổi Quyền và Chủ sở hữu

Để áp dụng các quyền `rwx` hoặc thay đổi chủ sở hữu file, quản trị viên sẽ sử dụng ba lệnh sau:

#### 3.1. `chown` (Change Owner)

Dùng để thay đổi chủ sở hữu (User Owner) và nhóm sở hữu (Group Owner) của file hoặc thư mục.

```bash
# Đổi chủ sở hữu thành 'alice'
chown alice file.txt

# Đổi cả chủ sở hữu ('alice') và nhóm ('devteam') cùng một lúc
chown alice:devteam file.txt
```

<img width="1192" height="171" alt="image" src="https://github.com/user-attachments/assets/42239ce7-c076-4e27-b43e-4a4b95e9ced5" />

#### 3.2. `chgrp` (Change Group)

Chỉ thay đổi nhóm sở hữu của file. Lệnh này tương đương với `chown :groupname` nhưng cú pháp ngắn gọn hơn khi chỉ cần thay đổi nhóm.

```bash
chgrp devteam file.txt
```

#### 3.3. `chmod` (Change Mode)

Dùng để thay đổi quyền truy cập `rwx` của file hoặc thư mục. Có hai cách sử dụng:

**Dùng ký tự (Symbolic mode):** Dễ đọc hơn khi cần thêm hoặc bớt một quyền cụ thể mà không ảnh hưởng đến các quyền còn lại.

```bash
chmod u+x script.sh    # Thêm quyền thực thi (x) cho User (chủ sở hữu)
chmod g-w file.txt     # Xóa quyền ghi (w) của Group
chmod o=r file.txt     # Gán chính xác chỉ quyền đọc (r) cho Others
```

**Dùng số (Octal mode):** Nhanh hơn khi muốn thiết lập lại toàn bộ quyền cùng một lúc. Mỗi quyền được biểu diễn bằng một con số: `r=4`, `w=2`, `x=1`.

```bash
chmod 755 script.sh    # rwxr-xr-x: chủ toàn quyền, group và others chỉ đọc và chạy
chmod 644 file.txt     # rw-r--r--: chủ đọc/ghi, group và others chỉ đọc
```

### 4. Quyền mặc định: Umask (User file-creation mode mask)

Khi tạo một file hay thư mục mới, hệ thống tự động áp dụng một bộ quyền mặc định. Cơ chế kiểm soát điều này gọi là **umask**.

**Nguyên lý hoạt động:**

- Quyền tối đa cho thư mục là `777`, cho file là `666`. Linux không mặc định cấp quyền thực thi (`x`) cho file mới nhằm tránh rủi ro bảo mật khi vô tình tạo ra các file có thể thi hành được.
- Quyền mặc định = Quyền tối đa **trừ đi** giá trị umask.

**Ví dụ với umask mặc định `022`:**

- Thư mục mới tạo: `777 - 022 = 755` (`rwxr-xr-x`)
- File mới tạo: `666 - 022 = 644` (`rw-r--r--`)

**Kiểm tra và thay đổi umask:**

```bash
umask              # Xem giá trị umask hiện tại
umask 027          # Thay đổi umask tạm thời trong phiên làm việc hiện tại
```

Để cấu hình umask cố định (áp dụng mỗi lần đăng nhập), cần thêm lệnh `umask 027` vào file `~/.bashrc` hoặc `/etc/profile`.

### 5. Quản lý đặc quyền: Root, `su` và `sudo`

Trong quản trị hệ thống, tài khoản có quyền lực tối cao là `root`. Việc quản lý cẩn thận tài khoản này và cách ủy quyền đặc quyền cho người dùng thường là một trong những yếu tố bảo mật quan trọng nhất.

**Tài khoản `root`:** Là superuser, có khả năng thực hiện bất kỳ thao tác nào trên hệ thống, vượt qua mọi giới hạn phân quyền `rwx` thông thường. Vì lý do đó, việc đăng nhập trực tiếp bằng `root` rất nguy hiểm và thường bị vô hiệu hóa trên các bản phân phối hiện đại như Ubuntu.

**`su` (Substitute User):** Cho phép chuyển đổi hoàn toàn sang một phiên làm việc của user khác, thường là root. Yêu cầu phải biết mật khẩu của user đích.

```bash
su -       # Chuyển sang root kèm theo toàn bộ biến môi trường của root
su - alice # Chuyển sang user alice
```

**`sudo` (Superuser DO):** Cho phép người dùng thường thực thi một lệnh cụ thể với quyền root, nhưng sử dụng **mật khẩu của chính người dùng đó**. Để sử dụng được `sudo`, người dùng phải được cấp quyền trong file `/etc/sudoers`.

```bash
sudo apt update    # Thực thi lệnh với quyền root
```

> **Lưu ý quan trọng:** Không bao giờ chỉnh sửa trực tiếp file `/etc/sudoers`. Phải dùng lệnh `visudo` để mở file này, vì `visudo` tự động kiểm tra lỗi cú pháp trước khi lưu, tránh khả năng làm hỏng cấu hình sudo toàn hệ thống.

### 6. Tường lửa (Firewall)

Tường lửa là phòng tuyến đầu tiên để bảo vệ máy chủ khỏi các truy cập mạng trái phép. Trên các hệ thống Linux như Ubuntu, **UFW (Uncomplicated Firewall)** là công cụ phổ biến và dễ sử dụng nhất để quản lý các quy tắc lọc gói tin của `iptables`.

Nguyên tắc cấu hình tường lửa cơ bản là: **"Từ chối tất cả các kết nối đến (Deny Incoming), chỉ cho phép các dịch vụ cần thiết (Allow)"**.

#### 6.1. Kiến trúc tường lửa trong Linux

Hệ thống tường lửa của Linux được thiết kế theo mô hình 3 lớp từ thấp đến cao:

**Lớp 1: Kernel Space — Netfilter (Lõi xử lý)**

Netfilter nằm sâu bên trong nhân (Kernel) của hệ điều hành Linux. Đây là "động cơ" thực sự của tường lửa — bao gồm các "hook" (điểm neo) được đặt trên đường đi của dòng dữ liệu mạng. Khi một gói tin đi vào, đi ra hoặc đi xuyên qua máy chủ, nó bắt buộc phải đi qua các hook này. Tại đây, Netfilter trực tiếp thực hiện việc cho phép (ACCEPT), loại bỏ (DROP), hoặc dịch địa chỉ (NAT) ở mức thấp nhất.

**Lớp 2: User Space — Low-level Backend (Công cụ tương tác cấp thấp)**

Vì Netfilter nằm trong Kernel nên con người không thể ra lệnh trực tiếp cho nó. Linux cung cấp các công cụ dòng lệnh để "dịch" các quy tắc xuống cho Netfilter:

- **`iptables`:** Công cụ kinh điển và nổi tiếng nhất. Cho phép quản trị viên định nghĩa các luồng (chains) và bảng (tables) để kiểm soát gói tin. Tuy nhiên, cú pháp của `iptables` rất phức tạp và dễ cấu hình sai.
- **`nftables`:** Công cụ thế hệ mới, được thiết kế để thay thế `iptables`. Cú pháp gọn gàng hơn, hiệu năng tốt hơn và hỗ trợ cả IPv4 lẫn IPv6 trong cùng một quy tắc.

**Lớp 3: User Space — Frontend Management (Công cụ quản lý cấp cao)**

Vì `iptables`/`nftables` quá phức tạp, các bản phân phối Linux đã tạo ra lớp Frontend cung cấp cú pháp đơn giản hơn, sau đó tự động biên dịch thành mã `iptables`/`nftables` phía dưới:

- **UFW (Uncomplicated Firewall):** Công cụ mặc định trên họ Debian/Ubuntu. Đúng với tên gọi, UFW cực kỳ đơn giản — chỉ cần gõ `ufw allow 80` thay vì một chuỗi lệnh dài của iptables.
- **Firewalld:** Công cụ mặc định trên họ RHEL/CentOS/Rocky Linux. Quản lý theo dạng vùng mạng (Zones) và cho phép áp dụng rule ngay lập tức mà không cần khởi động lại dịch vụ tường lửa.

#### 6.2. Cài đặt và cấu hình UFW

> **Lưu ý:** Chỉ nên cài đặt UFW trên hệ thống Linux chưa có bất kỳ control panel nào như cPanel, aaPanel, CyberPanel, vì các control panel này đều cài sẵn tiện ích tường lửa riêng. Cài đặt thêm UFW sẽ gây xung đột hệ thống.

**Bước 1: Cài đặt UFW**

```bash

sudo apt update
sudo apt upgrade -y

# Kiểm tra xem UFW đã được cài sẵn chưa
which ufw

# Nếu chưa có, tiến hành cài đặt
sudo apt install ufw

# Kiểm tra trạng thái — mặc định sau cài đặt sẽ là inactive
sudo ufw status
```

#### 6.3. Các thao tác quản lý UFW thường dùng

**Mở cổng theo dải port:**

```bash
sudo ufw allow 6000:6007/tcp
sudo ufw allow 6000:6007/udp
```

**Cho phép kết nối theo địa chỉ IP:**

```bash
# Cho phép một IP cụ thể kết nối vào mọi cổng
sudo ufw allow from 123.123.123.123

# Cho phép một IP cụ thể chỉ kết nối vào cổng 22 (SSH)
sudo ufw allow from 123.123.123.123 to any port 22
```

**Cho phép kết nối theo Subnet:**

```bash
# Cho phép toàn bộ dải IP 10.0.1.1 đến 10.0.1.254
sudo ufw allow from 10.0.1.0/24
```

**Chặn kết nối:**

```bash
# Chặn kết nối HTTP từ bên ngoài vào
sudo ufw deny http

# Chặn một địa chỉ IP cụ thể
sudo ufw deny from 123.123.123.123

# Chặn cổng 25 đi ra ngoài (ngăn dịch vụ SMTP gửi mail — thường dùng để chống spam)
sudo ufw deny out 25
```

**Xóa rule:**

```bash
# Xóa rule theo tên dịch vụ
sudo ufw delete allow http

# Xóa rule theo số port
sudo ufw delete allow 80
```

**Vô hiệu hóa UFW:**

```bash
sudo ufw disable
```

### 7. Giám sát hệ thống (Monitoring)

Giám sát là công cụ không thể thiếu của quản trị viên. Trong góc độ bảo mật và vận hành, giám sát giúp phát hiện các vấn đề sau:

- **Quá tải tài nguyên (CPU/RAM/Disk I/O):** Có thể là dấu hiệu của cuộc tấn công từ chối dịch vụ (DDoS) hoặc tiến trình mã độc đào coin (Cryptominer) đang chạy ngầm. Trọng tâm không chỉ nhìn vào % CPU mà phải nhìn vào **Load Average** (Tải trung bình).
- **Hết dung lượng ổ cứng:** Khi disk đầy 100%, các dịch vụ như Database (MySQL) hoặc Web Server sẽ ngừng hoạt động ngay lập tức vì không thể ghi log.
- **Dấu hiệu mạng bất thường:** Kiểm tra xem có port lạ nào đang mở và âm thầm kết nối ra ngoài (dấu hiệu của Backdoor hoặc C2 Server) hay không.

#### 7.1. Công cụ dòng lệnh trực tiếp (Local CLI Tools)

Đây là các công cụ cài đặt trực tiếp trên server, dùng để xử lý sự cố (troubleshooting) tức thời khi quản trị viên đang SSH vào máy.

#### 7.2. Hệ thống giám sát tập trung truyền thống (IT Infrastructure Monitoring)

Khi quản lý từ vài chục đến hàng nghìn server, quản trị viên không thể SSH vào từng máy để gõ lệnh. Lúc này cần các hệ thống cài đặt Agent (tác nhân thu thập dữ liệu) trên client và đẩy về server trung tâm.

- **Zabbix:** Giải pháp mã nguồn mở cực kỳ phổ biến trong doanh nghiệp. Rất mạnh trong việc giám sát hạ tầng vật lý, máy ảo (VM), thiết bị mạng (Switch/Router qua SNMP). Có hệ thống cảnh báo (Trigger/Alert) qua Email, Telegram rất chi tiết.
- **Nagios:** Hệ thống giám sát lâu đời, rất ổn định nhưng giao diện cũ và khó cấu hình hơn Zabbix. Hiện nay thường chỉ còn thấy ở các hệ thống cũ (Legacy).
- **PRTG Network Monitor:** Công cụ thương mại (có bản miễn phí giới hạn), giao diện đẹp, chuyên dùng để giám sát Network và Windows Server, cũng hỗ trợ Linux qua SNMP/SSH.

#### 7.3. Hệ thống giám sát hiện đại (Modern / Cloud-Native Stack)

Với xu hướng DevOps, Container (Docker/Kubernetes) và Microservices, các công cụ giám sát cũng tiến hóa thành các nền tảng phân tích dữ liệu chuỗi thời gian (Time-series data) và Log tập trung.

- **Prometheus & Grafana:** Prometheus đóng vai trò là "Database" thu thập các chỉ số (Metrics) từ các server thông qua `node_exporter`. Grafana đóng vai trò là "Giao diện" kết nối với Prometheus để vẽ ra các dashboard trực quan. *(Chi tiết đã trình bày trong báo cáo ngày 15/4)*

- **ELK Stack (Elasticsearch, Logstash, Kibana):** Chuyên dụng cho Giám sát Log (Log Management). Thay vì đọc file `/var/log` thủ công, ELK gom toàn bộ log của tất cả server về một chỗ, phân tích, tìm kiếm và vẽ biểu đồ cảnh báo — ví dụ: đếm số lần đăng nhập SSH thất bại trên toàn cụm server.

### 8. Quản lý cập nhật hệ thống và Vòng đời phiên bản (Patch Management & Release Cycle)

Trong môi trường doanh nghiệp, việc cập nhật hệ thống không diễn ra ngẫu hứng mà phải tuân thủ chiến lược vòng đời của hệ điều hành và nguyên tắc quản lý gói phần mềm.

#### 8.1. Vòng đời phiên bản Ubuntu (Release Cycle)

Ubuntu được phát hành theo một chu kỳ chặt chẽ, chia làm hai nhánh chính. Việc chọn đúng nhánh quyết định sự ổn định và tuổi thọ của các hệ thống máy chủ.

**Bản LTS (Long Term Support — Hỗ trợ dài hạn):**

- Phát hành 2 năm một lần, luôn vào tháng 4 năm chẵn (ví dụ: 20.04, 22.04, 24.04).
- Vòng đời hỗ trợ: **5 năm** (có thể mở rộng lên 10 năm với Ubuntu Pro).
- Đây là tiêu chuẩn bắt buộc cho môi trường Production của các doanh nghiệp, ưu tiên sự ổn định tuyệt đối thay vì các tính năng mới nhất.

**Bản Interim (Phát hành thông thường / Ngắn hạn):**

- Phát hành 6 tháng một lần vào tháng 4 năm lẻ và tháng 10 hàng năm (ví dụ: 23.10, 24.10, 25.04).
- Vòng đời hỗ trợ: Chỉ **9 tháng**.
- Dành cho nhà phát triển muốn trải nghiệm kernel hoặc công nghệ mới nhất. Tuyệt đối không dùng bản này cho máy chủ chạy dịch vụ thực tế vì sẽ rất nhanh bị ngừng hỗ trợ bảo mật.

#### 8.2. Cơ chế hoạt động của APT (Advanced Package Tool)

Trên Ubuntu và họ Debian, phần mềm được quản lý thông qua **APT** kết nối với các **Repositories (Kho phần mềm chính thức)**. Danh sách các kho này được lưu tại `/etc/apt/sources.list`.

Hai lệnh cốt lõi cần phân biệt rõ:

- **`apt update` (Cập nhật danh sách):** Lệnh này **không** cài đặt hay nâng cấp bất kỳ phần mềm nào. Nó chỉ tải về máy một "bản danh sách" mới nhất từ kho, để hệ thống biết phần mềm nào đang có phiên bản mới.
- **`apt upgrade` (Nâng cấp thực tế):** Dựa vào danh sách vừa tải, lệnh này mới thực sự tải các gói dữ liệu về máy và tiến hành cài đặt đè lên phiên bản cũ.

#### 8.3. Quy trình cập nhật hệ thống an toàn

**Bước 1: Kiểm tra phiên bản hệ điều hành hiện tại**

Trước khi làm gì, quản trị viên cần xác định phiên bản hệ điều hành đang chạy.

```bash
lsb_release -a
cat /etc/os-release
```

**Bước 2: Cập nhật danh sách gói và rà soát**

```bash
# Tải về danh sách gói mới nhất từ kho
sudo apt update

# Xem danh sách các gói có thể nâng cấp
apt list --upgradable
```

<img width="507" height="92" alt="image" src="https://github.com/user-attachments/assets/f059210d-8220-4a78-a429-935d450a07cc" />

*(Hệ thống đã được cập nhật trước đó nên danh sách gói cần nâng cấp rất ít)*

**Bước 3: Nâng cấp hệ thống và dọn dẹp**

```bash
# Tiến hành nâng cấp các gói phần mềm (tham số -y để tự động đồng ý)
sudo apt upgrade -y

# Xóa các gói phụ thuộc cũ không còn được sử dụng
```

**Bước 4: Nâng cấp toàn diện (Tùy chọn nâng cao)**

Khi muốn nâng cấp toàn bộ hệ điều hành từ phiên bản cũ sang mới (ví dụ từ Ubuntu 22.04 lên 24.04), lệnh `upgrade` thông thường sẽ bỏ qua các gói phức tạp để tránh gây lỗi. Cần dùng các lệnh chuyên biệt hơn:

```bash
# Nâng cấp toàn diện, tự động xử lý các xung đột gói
sudo apt full-upgrade

# Nâng cấp hẳn phiên bản hệ điều hành (Release Upgrade)
# Cực kỳ cẩn thận khi chạy lệnh này trên môi trường Production
sudo do-release-upgrade
```

### 9. Phân tích Log để phát hiện tấn công

Trong an toàn thông tin, nếu hệ thống giám sát báo cho biết *khi nào* hệ thống có vấn đề, thì phân tích Log sẽ trả lời cho câu hỏi *ai đã làm điều đó* và *làm như thế nào*. Mọi dấu vết của kẻ tấn công — dù thành công hay thất bại — đều được hệ điều hành ghi nhận lại.

#### 9.1. Các file log quan trọng

Trên Ubuntu/Debian, log được lưu tập trung tại `/var/log/`:

- **`/var/log/auth.log`:** Ghi lại toàn bộ các sự kiện xác thực — đăng nhập SSH, chuyển quyền `su`, thực thi lệnh bằng `sudo`. Đây là nguồn thông tin chính để phát hiện các cuộc tấn công dò mật khẩu (Brute-force).
- **`/var/log/syslog`** (hoặc dùng `journalctl`): Ghi lại các thông điệp chung của hệ điều hành và các dịch vụ nền.
- **Log của dịch vụ ứng dụng** (ví dụ: `/var/log/apache2/access.log`): Ghi nhận mọi request truy cập từ bên ngoài. Phân tích log này giúp phát hiện các cuộc tấn công vào lớp ứng dụng web như SQL Injection hay Directory Traversal.

#### 9.2. Thực hành phân tích log SSH

Khi một IP lạ liên tục thử các mật khẩu khác nhau (Brute-force attack), file `auth.log` sẽ ghi lại hàng loạt dòng báo lỗi. Quản trị viên sử dụng kết hợp các công cụ `grep`, `awk`, `sort`, `uniq` để trích xuất thông tin cần thiết từ hàng vạn dòng log.

```bash
# Đọc 20 dòng cuối của log xác thực để xem cấu trúc
sudo tail -n 20 /var/log/auth.log

# Lọc và hiển thị tất cả các lần đăng nhập thất bại
sudo grep "Failed password" /var/log/auth.log
```

### 1. Windows Event Logs và Event Viewer

#### 1.1. Tổng quan về Windows Event Logs

Trong điều tra số (Digital Forensics), Windows Event Logs là một trong những nguồn thông tin quan trọng nhất. Event Logs ghi lại các sự kiện liên quan đến software, hardware, hệ điều hành và bảo mật. Service **Windows Event Log** chịu trách nhiệm thu thập các sự kiện từ nhiều nguồn khác nhau và lưu trữ tập trung tại:

```
C:\Windows\System32\winevt\Logs
```

<img width="1580" height="928" alt="image" src="https://github.com/user-attachments/assets/2787d0ec-db5a-45e6-b149-85bfd5c036a6" />

Các sự kiện này giúp điều tra viên biết được chuyện gì đã xảy ra, thời gian xảy ra, người dùng liên quan và các tài nguyên, hệ thống được truy cập.

#### 1.2. Phân loại theo Nguồn ghi (Log Types)

Windows phân chia thành các nhật ký tiêu chuẩn và bản ghi chuyên biệt:

- **Application (Ứng dụng):** Chứa các sự kiện được ghi bởi phần mềm/ứng dụng cài đặt trên máy. Ví dụ: ứng dụng cơ sở dữ liệu ghi lại lỗi không thể đọc file.
- **Security (Bảo mật):** Ghi lại các sự kiện dựa trên chính sách kiểm toán (audit policies) — bao gồm các nỗ lực đăng nhập hợp lệ/không hợp lệ, thao tác tạo/mở/xóa file. **Đây là file log quan trọng nhất đối với System/Security Engineer.**
- **System (Hệ thống):** Chứa các sự kiện do các thành phần cốt lõi của Windows tạo ra, ví dụ: lỗi driver hoặc service thất bại khi khởi động.
- **CustomLog (Log tùy chỉnh):** Chứa sự kiện do ứng dụng tự tạo ra nhật ký riêng biệt để kiểm soát kích thước file log hoặc gán quyền bảo mật (ACL) riêng. Ví dụ: log của PowerShell, Task Scheduler.

#### 1.3. Phân loại theo Mức độ nghiêm trọng (Event Types)

Mỗi dòng log khi sinh ra đều được gán nhãn để quản trị viên biết ngay bản chất của sự kiện:

<img width="599" height="212" alt="image" src="https://github.com/user-attachments/assets/c68ce832-699c-49d7-bf04-707d1dbb827a" />

- **Error (Lỗi):** Sự kiện chỉ ra vấn đề nghiêm trọng như mất dữ liệu hoặc mất chức năng. Ví dụ: một service quan trọng không thể khởi động.
- **Warning (Cảnh báo):** Sự kiện không nghiêm trọng ngay lập tức nhưng cảnh báo về vấn đề có thể xảy ra trong tương lai. Ví dụ: dung lượng ổ đĩa đang ở mức thấp.

### 2. Task Scheduler

Trong môi trường Windows Server, quản trị viên hiếm khi tự tay chạy các script bảo trì hàng ngày. Mọi tác vụ từ sao lưu cơ sở dữ liệu, dọn dẹp ổ đĩa, đến khởi động lại dịch vụ vào ban đêm đều được giao cho **Task Scheduler** thực hiện tự động.

#### 2.1. Cấu thành của một Task

Một tác vụ hoàn chỉnh bao gồm ba thành phần cốt lõi:

- **Triggers (Trình kích hoạt):** Quyết định *khi nào* tác vụ sẽ chạy. Có thể theo thời gian (hàng ngày lúc 2h sáng), theo trạng thái (khi máy khởi động), hoặc rất mạnh mẽ: **theo sự kiện (On an Event)** — ví dụ: kích hoạt script chặn IP ngay khi Event Viewer ghi nhận Event ID 4625 (Brute-force attack).
- **Actions (Hành động):** Quyết định *làm gì*. Thường là khởi chạy một phần mềm, file batch (`.bat`) hoặc script PowerShell (`.ps1`).
- **Security Context (Bối cảnh bảo mật):** Quyết định *chạy bằng quyền của ai*. Các script hệ thống thường phải chạy với quyền **SYSTEM** để hoạt động ổn định kể cả khi quản trị viên đã đăng xuất.

### 3. Windows Firewall

Windows Firewall không chỉ là công cụ bật/tắt đơn giản. Ở cấp độ quản trị Windows Server, đây là một hệ thống kiểm soát luồng dữ liệu hai chiều với ba khái niệm cốt lõi mà System Engineer phải nắm vững.

#### 3.1. Các khái niệm cốt lõi

**Luồng dữ liệu (Traffic Direction):**

- **Inbound Rules (Quy tắc kết nối vào):** Kiểm soát những ai từ bên ngoài được phép truy cập vào máy chủ. Mặc định: **CHẶN TẤT CẢ (Block all)**.
- **Outbound Rules (Quy tắc kết nối ra):** Kiểm soát việc máy chủ được phép truy cập ra bên ngoài. Mặc định: **CHO PHÉP TẤT CẢ (Allow all)**.

**Network Profiles (Hồ sơ mạng):** Windows tự động áp dụng các bộ quy tắc khác nhau tùy theo mạng đang kết nối:

- **Domain Profile:** Khi máy tính gia nhập mạng nội bộ công ty (Active Directory).
- **Private Profile:** Mạng nội bộ an toàn (mạng LAN ở nhà hoặc phòng lab).
- **Public Profile:** Mạng công cộng không an toàn (Wi-Fi quán cafe hoặc kết nối trực tiếp ra Internet).

**Actions (Hành động xử lý):** Allow (Cho phép), Block (Chặn), hoặc Allow if secure (Chỉ cho phép nếu kết nối được mã hóa bằng IPsec).

#### 3.2. Kiểm tra trạng thái Firewall

Mở cửa sổ **Run** (phím `Win + R`), nhập `control firewall.cpl` và nhấn Enter. Trạng thái tường lửa được hiển thị theo màu sắc cho từng Network Profile (Xanh lá: đang bật, Đỏ: đang tắt).

### 4. Quản lý Phân quyền File và Thư mục (NTFS Permissions)

Windows sử dụng cơ chế **NTFS Permissions** thay vì mô hình đơn giản của Linux. Thay vì chỉ chia làm 3 nhóm cố định (User, Group, Others), Windows sử dụng **ACL (Access Control List)** cho phép gán quyền cho vô số người dùng hoặc nhóm khác nhau trên cùng một file.

#### 4.1. Các mức quyền cơ bản

- **Full Control (Toàn quyền):** Có thể làm mọi thứ, bao gồm thay đổi quyền của người khác và chiếm quyền sở hữu (Take Ownership). Chỉ nên cấp cho Administrator.
- **Modify (Sửa đổi):** Đọc, ghi, xóa file và thư mục, nhưng không thể thay đổi quyền hay chuyển quyền sở hữu. Đây là mức quyền lý tưởng để cấp cho nhân viên làm việc thông thường.
- **Read & Execute (Đọc và thực thi):** Xem nội dung file và chạy được các ứng dụng/script.
- **List Folder Contents (Liệt kê nội dung):** Chỉ áp dụng cho thư mục, cho phép xem bên trong có các file nào.
- **Read (Đọc) / Write (Ghi):** Mức quyền thấp nhất để xem hoặc tạo/chỉnh sửa nội dung.

#### 4.2. Hai nguyên tắc quan trọng trong NTFS

- **Tính kế thừa (Inheritance):** Mặc định, mọi file và thư mục con sẽ tự động nhận y hệt các quyền từ thư mục cha. Điều này đơn giản hóa quản lý nhưng cần chú ý khi muốn đặt quyền riêng biệt.
- **Deny luôn thắng Allow:** Nếu User A thuộc nhóm được cấp quyền `Allow Read`, nhưng bạn lại đánh dấu `Deny Read` trực tiếp cho User A, thì kết quả là User A sẽ **bị chặn hoàn toàn**. Windows luôn ưu tiên lệnh cấm (Deny) hơn lệnh cho phép (Allow).

#### 4.3. Thực hành: Phân quyền thư mục cho phòng Kế toán

**Bước 1:** Tạo thư mục `Data_KeToan` tại ổ C. Click chuột phải vào thư mục → chọn **Properties** → tab **Security**.

<img width="1067" height="832" alt="image" src="https://github.com/user-attachments/assets/0dd0289b-3150-44e4-8071-079e78d9e7a6" />

**Bước 2: Ngắt kế thừa (Disable Inheritance)**

Mặc định, thư mục đang kế thừa quyền từ thư mục cha (bao gồm nhóm `Users` — tức là tất cả mọi người đều có quyền đọc). Để kiểm soát chặt hơn, cần ngắt kế thừa trước.

Bấm **Advanced** ở góc dưới tab Security.

<img width="1106" height="781" alt="image" src="https://github.com/user-attachments/assets/dfe102df-be6b-4e4d-8564-88efbe3e78eb" />

Bấm **Disable inheritance**.

<img width="788" height="441" alt="image" src="https://github.com/user-attachments/assets/7d988236-30f9-4d79-801a-40bb541cf608" />

Chọn: **Convert inherited permissions into explicit permissions on this object** — chuyển các quyền đang kế thừa thành quyền độc lập để có thể tự do chỉnh sửa.

<img width="1145" height="794" alt="image" src="https://github.com/user-attachments/assets/41b73d9d-6151-44dd-8360-be7403df8b6a" />

**Bước 3: Dọn dẹp quyền cũ và cấp quyền mới**

Xóa các nhóm không liên quan (như `Users` hoặc `Authenticated Users`) bằng cách chọn và bấm **Remove**. Chỉ giữ lại `SYSTEM` và `Administrators`.

Bấm **Add** → chọn **Select a principal** → gõ tên user hoặc nhóm (ví dụ: `ketoan_alice`) → bấm **Check Names** để xác nhận → bấm **OK**.

## V. So sánh Linux và Windows: Bảo mật, Logs và Firewall

### 1. Quản lý Người dùng và Phân quyền

| Tiêu chí | Linux | Windows |
|---|---|---|
| Mô hình phân quyền | Dựa trên ba nhóm cố định: Owner (chủ), Group (nhóm), Others (người khác) với ba quyền r/w/x | Dựa trên ACL (Access Control List) — gán quyền chi tiết cho từng user hoặc nhóm riêng biệt |
| Công cụ quản lý | Lệnh dòng lệnh: `useradd`, `usermod`, `passwd`, `chmod`, `chown` | Giao diện đồ họa: Local Users and Groups (lusrmgr.msc) hoặc lệnh `net user` |
| Tài khoản đặc quyền | `root` — có toàn quyền hệ thống, truy cập gián tiếp qua `sudo` | `Administrator` — tương đương root, nhưng trên Windows Server thường dùng Role-Based Access Control (RBAC) qua Active Directory |
| Ủy quyền đặc quyền | `sudo` với cấu hình trong `/etc/sudoers` — có thể phân quyền rất chi tiết từng lệnh | UAC (User Account Control) nhắc xác nhận khi thực thi với quyền Admin; cấu hình phân quyền qua Group Policy |
| Kế thừa quyền | Không có kế thừa tự động — quyền được gán trực tiếp, thư mục con không tự thừa kế | Thừa kế tự động từ thư mục cha (Inheritance) — có thể ngắt bằng "Disable Inheritance" |
| Đặc quyền đặc biệt | Sticky Bit, SUID, SGID | Không có tương đương trực tiếp — dùng Advanced Security Settings |

### 2. Hệ thống Log

| Tiêu chí | Linux | Windows |
|---|---|---|
| Vị trí lưu trữ | Thư mục `/var/log/` — file text thuần (plain text), dễ đọc bằng bất kỳ công cụ nào | `C:\Windows\System32\winevt\Logs\` — file nhị phân định dạng `.evtx`, cần công cụ chuyên biệt để đọc |
| Định dạng | Plain text — có thể dùng `grep`, `awk`, `tail`, `cat` để phân tích trực tiếp | Binary `.evtx` — cần dùng Event Viewer (GUI) hoặc PowerShell (`Get-WinEvent`) để truy xuất |
| Phân loại log | Theo chức năng: `auth.log` (xác thực), `syslog` (hệ thống), `kern.log` (kernel), log riêng của từng ứng dụng | Theo nguồn: Application, Security, System, Custom logs |
| Phân loại mức độ | Thường phân biệt qua từ khóa trong nội dung: INFO, WARNING, ERROR, CRITICAL | Phân loại chính thức: Information, Warning, Error, Success Audit, Failure Audit |
| Log bảo mật | `/var/log/auth.log` — ghi nhận SSH, sudo, đăng nhập thất bại | Security Log — ghi nhận chi tiết qua Event ID (4624: đăng nhập thành công, 4625: thất bại, 4648: đăng nhập bằng credentials khác) |
| Truy cập log | Cần quyền `root` hoặc `sudo` để đọc `/var/log/auth.log` và `/var/log/syslog` | Cần quyền Administrator để đọc Security Log |
| Công cụ phân tích | `grep`, `awk`, `sort`, `uniq`, `tail`, `journalctl` — mạnh cho xử lý pipeline | Event Viewer (GUI), `wevtutil`, `Get-WinEvent` (PowerShell), Task Scheduler kết hợp Event ID |
| Tự động hóa theo log | Phải viết script thủ công hoặc dùng `fail2ban` | Task Scheduler hỗ trợ trigger "On an Event" — tự động kích hoạt hành động khi một Event ID cụ thể xuất hiện |

### 3. Tường lửa (Firewall)

| Tiêu chí | Linux (UFW) | Windows Firewall |
|---|---|---|
| Lớp nền tảng | Netfilter (trong Kernel) → `iptables`/`nftables` → UFW (frontend) | Windows Filtering Platform (WFP) tích hợp sâu vào kernel Windows |
| Công cụ quản lý | Dòng lệnh: `ufw allow`, `ufw deny`, `ufw status` | Giao diện đồ họa: `control firewall.cpl` và **Windows Firewall with Advanced Security** (wf.msc) |
| Chính sách mặc định | Sau khi cấu hình: Deny Incoming, Allow Outgoing | Inbound: Block all (mặc định); Outbound: Allow all (mặc định) |
| Hồ sơ mạng | Không phân biệt theo profile — áp dụng rule cho tất cả giao diện mạng | Ba profile riêng biệt: Domain, Private, Public — áp dụng quy tắc khác nhau tùy mạng |
| Cấu hình theo IP | Hỗ trợ đầy đủ: `ufw allow from IP`, `ufw deny from IP`, hỗ trợ CIDR notation | Hỗ trợ đầy đủ: Local IP và Remote IP trong mục Scope của từng rule |
| Ghi log | Dùng `ufw logging on` để bật; log ghi vào `/var/log/ufw.log` | Có thể bật logging trong **Advanced Settings** → Properties → Logging |
| Backup cấu hình | Không có tính năng export/import chính thức — phải sao lưu thủ công file `/etc/ufw/` | Hỗ trợ **Export Policy** / **Import Policy** trực tiếp trong giao diện |
| Độ phức tạp cấu hình | Đơn giản cho người mới; phức tạp khi cần rule nâng cao (phải xuống `iptables` trực tiếp) | Giao diện đồ họa trực quan nhưng nhiều tùy chọn hơn; hỗ trợ IPsec, rule theo ứng dụng |

### 4. Tự động hóa tác vụ

| Tiêu chí | Linux | Windows |
|---|---|---|
| Công cụ chính | `cron` / `crontab` — lập lịch theo thời gian; `systemd timers` — thế hệ mới hơn | Task Scheduler — hỗ trợ nhiều loại trigger phong phú hơn |
| Trigger theo thời gian | Hỗ trợ (cú pháp cron: phút, giờ, ngày, tháng, thứ) | Hỗ trợ (giao diện trực quan, dễ thiết lập hơn) |
| Trigger theo sự kiện | Hạn chế — phải kết hợp với script giám sát log (`fail2ban`, `logwatch`) | Hỗ trợ tích hợp sẵn: "On an Event" — kích hoạt theo Event ID cụ thể trong Event Viewer |
| Trigger theo trạng thái hệ thống | Hỗ trợ qua `systemd` (khi service khởi động, khi network sẵn sàng) | Hỗ trợ: khi khởi động máy, khi đăng nhập, khi máy rảnh |
| Ngữ cảnh thực thi | Theo user hoặc root — cấu hình trong `crontab -e` hoặc `/etc/cron.d/` | Cấu hình **Security Context** riêng — có thể chạy với quyền SYSTEM kể cả khi user đã đăng xuất |
| Giao diện | Dòng lệnh thuần — phải nhớ cú pháp cron | Giao diện đồ họa đầy đủ — trực quan, dễ quản lý hơn |

### 5. Nhận xét tổng quan

Cả Linux và Windows đều cung cấp đầy đủ các cơ chế bảo mật cốt lõi cần thiết cho quản trị hệ thống, nhưng triết lý thiết kế và cách tiếp cận khác nhau rõ rệt:

**Linux** ưu tiên tính linh hoạt và kiểm soát chi tiết qua dòng lệnh. Mọi thứ đều có thể tự động hóa và tích hợp vào pipeline của shell script. Log được lưu dưới dạng text thuần, dễ phân tích bằng các công cụ chuẩn UNIX. Phù hợp với các quản trị viên có kinh nghiệm dòng lệnh và môi trường server đòi hỏi tùy biến cao.

**Windows** ưu tiên tính dễ sử dụng và tích hợp chặt chẽ giữa các thành phần. Giao diện đồ họa trực quan giúp người mới tiếp cận nhanh hơn. Hệ thống Event Viewer với Event ID có cấu trúc rõ ràng, và Task Scheduler với trigger theo sự kiện là những điểm mạnh nổi bật so với Linux. Phù hợp với môi trường doanh nghiệp sử dụng hệ sinh thái Microsoft (Active Directory, Group Policy, Exchange).
