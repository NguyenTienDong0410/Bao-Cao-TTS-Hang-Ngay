## I. Monitor Prometheus + Grafana

### 1. Grafana Là Gì?

Grafana là một ứng dụng web mã nguồn mở cho phép phân tích và trực quan hóa dữ liệu, thiết kế các bảng điều khiển tương tác. Grafana hỗ trợ truy vấn dữ liệu và tạo các biểu đồ, dashboard từ nhiều nguồn dữ liệu khác nhau, giúp theo dõi và phân tích hiệu suất hệ thống dễ dàng hơn.

**Grafana được sử dụng để:**

- Tạo các dashboard tương tác và trực quan hóa dữ liệu (tương tự Tableau).
- Giám sát thời gian thực nhiều nguồn dữ liệu, bao gồm cả cơ sở hạ tầng Kubernetes.
- Phân tích hiệu suất hệ thống chi tiết.
- Thiết lập cảnh báo tự động dựa trên các ngưỡng được định nghĩa trước, thông báo về sự cố trước khi chúng leo thang.

## II. Cài Đặt Prometheus Trên Ubuntu 22.04

### Prometheus Là Gì?

Prometheus là hệ thống giám sát và cảnh báo mã nguồn mở dành cho các hệ thống phân tán và Cloud Native. Điểm khác biệt của Prometheus nằm ở mô hình **Pull-based**: chủ động thu thập dữ liệu từ các đối tượng giám sát qua HTTP endpoint và lưu trữ dưới dạng **Time-series** (chuỗi thời gian).

Prometheus cung cấp ngôn ngữ truy vấn **PromQL** linh hoạt, cho phép phân tích dữ liệu chuyên sâu và thiết lập các quy tắc cảnh báo tự động. Với khả năng tích hợp sâu vào hệ sinh thái Docker, Kubernetes và Grafana, Prometheus là tiêu chuẩn hàng đầu trong quản trị và theo dõi hiệu năng hạ tầng hiện đại.

![Kiến trúc tổng quan của hệ thống giám sát Prometheus](https://github.com/user-attachments/assets/10ee9842-b71a-4c5c-b536-c574736fb4bd)
*Hình 9: Kiến trúc Prometheus với mô hình Pull-based – Prometheus chủ động scrape metrics từ các Exporter, lưu vào TSDB (Time-Series Database), sau đó Grafana truy vấn để hiển thị.*

## III. Tổng Quan Về PromQL

**PromQL (Prometheus Query Language)** là ngôn ngữ truy vấn được thiết kế chuyên biệt cho Prometheus. Khác với SQL (dùng cho cơ sở dữ liệu quan hệ), PromQL được tối ưu hóa để xử lý và phân tích **dữ liệu chuỗi thời gian (Time-Series Data)**. PromQL là ngôn ngữ thuần chức năng (functional) và hoạt động ở chế độ **chỉ đọc (read-only)**.

**Vai trò cốt lõi của PromQL:**

- **Truy xuất dữ liệu đa chiều:** Cô lập và trích xuất chính xác các luồng số liệu từ hàng nghìn thiết bị, máy chủ, hoặc zone mạng khác nhau.
- **Hai kiểu dữ liệu đầu ra chính:**
  - **Instant Vector:** Trả về tập hợp các chuỗi thời gian chứa giá trị mới nhất tại một thời điểm duy nhất.
  - **Range Vector:** Trả về mảng dữ liệu trong một khoảng thời gian xác định (ví dụ: 5 phút qua).
- **Nền tảng của Cảnh báo và Trực quan hóa:** PromQL là ngôn ngữ để xây dựng biểu đồ trên Grafana và định nghĩa các quy tắc cảnh báo (Alerting Rules).

PromQL tuân theo cú pháp biểu thức chính quy **Re2** của Google.

### Cú Pháp Cơ Bản

```
metric_name{label="value"}[range]
```

**Ví dụ:**

```promql
http_requests_total{job="apiserver", handler="/api/comments"}[5m]
```

Truy vấn số lượng request HTTP của job `apiserver` tới handler `/api/comments` trong 5 phút gần nhất. Nếu bỏ `[5m]`, kết quả trả về là **Instant Vector** (giá trị tại thời điểm hiện tại).

### Truy Vấn Con (Subquery)

```promql
rate(http_requests_total{job="apiserver", handler="/api/comments"}[5m])[range:resolution]
```

Hàm `rate()` tính tốc độ tăng trung bình trên khoảng thời gian 5m. Phần `[range:resolution]` kiểm soát độ phân giải khi hiển thị đồ thị.

> **Lưu ý:** Hạn chế lồng truy vấn quá sâu (nested queries) vì làm tăng độ phức tạp và khó bảo trì.

### Hàm Tích Hợp – Ví Dụ `label_join()`

```promql
label_join(up{job="api-server", src1="a", src2="b", src3="c", srcN="z"},
           "New", ",", "src1", "src2", "src3", "srcN")
```

Kết hợp nhiều label thành một label mới (`New`) với giá trị `a,b,c,z`, phân cách bằng dấu phẩy.

## IV. Monitor Server Linux – Node Exporter

Node Exporter là công cụ thu thập metrics từ hệ thống Linux (CPU, RAM, Disk, Network,...) và expose qua HTTP endpoint để Prometheus scrape.

### Cài Đặt Node Exporter

```bash
# Tải ứng dụng
wget https://github.com/prometheus/node_exporter/releases/download/v1.11.1/node_exporter-1.11.1.linux-amd64.tar.gz

# Giải nén
tar -xzf node_exporter-1.11.1.linux-amd64.tar.gz

# Xóa tệp tải về
rm node_exporter-1.11.1.linux-amd64.tar.gz

# Đổi tên thư mục
mv node_exporter-1.11.1.linux-amd64 node_exporter-1.11.1

# Vào thư mục ứng dụng
cd node_exporter-1.11.1

# Kiểm tra phiên bản
./node_exporter --version

# Chạy ứng dụng
./node_exporter
```

Node Exporter chạy ở cổng **9100** theo mặc định, đường dẫn expose metrics là `/metrics`.

![Node Exporter khởi động thành công và đang lắng nghe tại cổng 9100](https://github.com/user-attachments/assets/fd5d7827-3238-4971-995d-2ae0ab792432)
*Hình 20: Node Exporter khởi động và lắng nghe tại `http://localhost:9100/metrics`. Log hiển thị các collector đang được kích hoạt (cpu, diskstats, filesystem, meminfo,...).*

### Cấu Hình Prometheus Scrape Node Exporter

Thêm job mới vào `/etc/prometheus/prometheus.yml`:

```bash
nano /etc/prometheus/prometheus.yml
```

```yaml
  - job_name: "node_exporter"
    static_configs:
      - targets: ["localhost:9100"]
```

![File prometheus.yml sau khi thêm job node_exporter](https://github.com/user-attachments/assets/37a23acd-7830-417e-aa97-b553ab33a02c)
*Hình 21: Cấu hình `prometheus.yml` với job `node_exporter` được thêm vào phần `scrape_configs`, trỏ tới `localhost:9100`.*

Khởi động lại Prometheus và kiểm tra trên giao diện web:

```bash
systemctl restart prometheus
```

### Bước 1: Cài Đặt mysqld_exporter


Tạo file cấu hình chứa thông tin đăng nhập MySQL:

```bash
nano /etc/.my.cnf
```

```ini
[client]
user=exporter
password=MatKhauKho123
```

### Bước 2: Tạo Systemd Service cho mysqld_exporter

```bash
nano /etc/systemd/system/mysqld_exporter.service
```

```ini
[Unit]
Description=Prometheus MySQL Exporter
After=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/mysqld_exporter \
  --config.my-cnf=/etc/.my.cnf
Restart=always

[Install]
WantedBy=multi-user.target
```

Khởi động dịch vụ:

```bash
systemctl daemon-reload
systemctl enable --now mysqld_exporter
```

mysqld_exporter chạy ở cổng **9104** theo mặc định.

## VII. Tổng Quan Về HAProxy

### HAProxy Là Gì?

**HAProxy (High Availability Proxy)** là công cụ mã nguồn mở cho giải pháp cân bằng tải TCP và HTTP. HAProxy được sử dụng để cải thiện hiệu năng và độ sẵn sàng của các ứng dụng web bằng cách phân phối lưu lượng truy cập qua nhiều máy chủ, giúp giảm thời gian phản hồi và tăng thông lượng. Các tổ chức lớn như GitHub, Stack Overflow, Reddit, Twitter và Amazon Web Services đều sử dụng HAProxy.

![Kiến trúc tổng quan HAProxy với nhiều backend server](https://github.com/user-attachments/assets/aa4c94a4-8746-4993-9721-8001af146a39)
*Hình 31: HAProxy đứng ở giữa, nhận request từ client và phân phối tới các backend server dựa trên thuật toán cân bằng tải được cấu hình.*

### Tính Năng Chính

- Cân bằng tải Layer 4 (TCP) và Layer 7 (HTTP).
- Hỗ trợ HTTP/1.1, HTTP/2, gRPC, FastCGI.
- Chấm dứt SSL/TLS.
- Lưu trữ chứng chỉ SSL động.
- Ghi log chi tiết.
- Xác thực HTTP.
- Rewrite URL.
- Đa luồng.

---

### Các Khái Niệm Quan Trọng

#### ACL (Access Control List)

ACL được dùng để kiểm tra điều kiện và thực hiện hành động (chọn server, chặn request) dựa trên kết quả kiểm tra.

**Ví dụ:**

```
acl url_blog path_beg /blog
```

ACL này khớp với mọi request có đường dẫn bắt đầu bằng `/blog`.

#### Backend

Backend là tập hợp các server nhận request được chuyển tiếp từ HAProxy. Một backend được định nghĩa bởi thuật toán cân bằng tải và danh sách server. Thêm nhiều server vào backend giúp tăng khả năng chịu tải và độ tin cậy.

**Ví dụ cấu hình backend:**

```
backend web-backend
    balance roundrobin
    server web1 web1.yourdomain.com:80 check
    server web2 web2.yourdomain.com:80 check

backend blog-backend
    balance roundrobin
    mode http
    server blog1 blog1.yourdomain.com:80 check
    server blog2 blog2.yourdomain.com:80 check
```

- `balance roundrobin`: Thuật toán cân bằng tải vòng tròn.
- `mode http`: Sử dụng Layer 7 proxy.
- `check`: Kích hoạt health check để phát hiện server bị lỗi.

#### Frontend

Frontend xác định cách request đến được chuyển tiếp tới backend, bao gồm: địa chỉ IP và cổng lắng nghe, các ACL, và các quy tắc `use_backend` / `default_backend`.

#### Sticky Sessions (Session Persistence)

Sticky Sessions là kỹ thuật gắn kết một client với một backend server cố định trong suốt phiên làm việc.

**Lý do cần Sticky Sessions:** Khi ứng dụng là **stateful** (lưu dữ liệu phiên trên server như login, giỏ hàng, PHP session), nếu không có sticky sessions, HAProxy sẽ phân phối request ngẫu nhiên sang server khác, dẫn đến mất session, đăng xuất không mong muốn, hoặc lỗi ứng dụng.

> **Khuyến nghị:** Sticky Sessions không phải giải pháp lâu dài. Giải pháp lý tưởng là thiết kế ứng dụng **stateless** và lưu session vào bộ nhớ tập trung như Redis hoặc Memcached.

---

### Các Thuật Toán Cân Bằng Tải

| Thuật toán | Mô tả |
|---|---|
| **Round Robin** | Phân phối request lần lượt đến các server theo vòng quay. |
| **Weighted Round Robin** | Như Round Robin nhưng mỗi server có trọng số (weight); server có trọng số cao hơn nhận nhiều request hơn. |
| **Least Connections** | Chuyển request đến server đang có ít kết nối đang hoạt động nhất. |
| **Least Response Time** | Chuyển request đến server có thời gian phản hồi nhanh nhất. |
| **IP Hash** | Dựa trên IP của client để luôn kết nối đến cùng một server backend. |

## VIII. Triển Khai HAProxy + Keepalived Cho Apache Trên Ubuntu 22.04

### Phần 1: Cấu Hình Apache (Web Backend)

**Thực hiện tại Node1 (Web1 – 10.10.11.86):**

```bash
sudo apt update
sudo apt install apache2 -y

# Đổi port từ 80 thành 8081 để nhường cổng 80 cho HAProxy
sudo sed -i 's/Listen 80/Listen 8081/' /etc/apache2/ports.conf
sudo sed -i 's/:80/:8081/' /etc/apache2/sites-enabled/000-default.conf

echo '<h1>Chào mừng tới Web1</h1>' | sudo tee /var/www/html/index.html

sudo systemctl start apache2
sudo systemctl enable apache2
```

**Thực hiện tại Node2 (Web2 – 10.10.11.87):**

```bash
sudo apt update
sudo apt install apache2 -y

sudo sed -i 's/Listen 80/Listen 8081/' /etc/apache2/ports.conf
sudo sed -i 's/:80/:8081/' /etc/apache2/sites-enabled/000-default.conf

echo '<h1>Chào mừng tới Web2</h1>' | sudo tee /var/www/html/index.html

sudo systemctl start apache2
sudo systemctl enable apache2
```

> **Lý do đổi port Apache sang 8081:** HAProxy sẽ lắng nghe ở cổng 80 và chuyển tiếp request đến Apache tại cổng 8081. Nếu không đổi port, HAProxy và Apache sẽ xung đột khi cùng cố chiếm cổng 80.

### Phần 2: Triển Khai Keepalived (Virtual IP)

Keepalived cung cấp một **Virtual IP (VIP)** dùng chung giữa hai node HAProxy. Khi node Master bị lỗi, VIP tự động chuyển sang node Backup, đảm bảo tính sẵn sàng cao (HA).

**Cài đặt trên cả 2 node:**

```bash
sudo apt update
sudo apt install keepalived -y
```

![Cài đặt Keepalived thành công trên node](https://github.com/user-attachments/assets/070788f4-386c-4720-9c8a-3cde21a0f35e)
*Hình 32: Keepalived được cài đặt thành công. Keepalived sử dụng giao thức VRRP (Virtual Router Redundancy Protocol) để quản lý VIP giữa các node.*

![Keepalived đang chạy và quản lý VIP](https://github.com/user-attachments/assets/ff45f4f2-154f-4034-b3cc-89f17d9905f2)
*Hình 33: Dịch vụ Keepalived hoạt động, node Master đang giữ VIP `10.10.10.89`.*

**Cấu hình Keepalived – Node1 (Master):**

```bash
sudo cat > /etc/keepalived/keepalived.conf << 'EOF'
vrrp_script chk_haproxy {
    script "killall -0 haproxy"
    interval 2
    weight 2
}

vrrp_instance VI_1 {
    interface enp2s1
    state MASTER
    virtual_router_id 51
    priority 101
    virtual_ipaddress {
        10.10.10.89/24
    }
    track_script {
        chk_haproxy
    }
}
EOF
```

**Cấu hình Keepalived – Node2 (Backup):**

```bash
sudo cat > /etc/keepalived/keepalived.conf << 'EOF'
vrrp_script chk_haproxy {
    script "killall -0 haproxy"
    interval 2
    weight 2
}

vrrp_instance VI_1 {
    interface enp2s1
    state BACKUP
    virtual_router_id 51
    priority 100
    virtual_ipaddress {
        10.10.10.89/24
    }
    track_script {
        chk_haproxy
    }
}
EOF
```

> **Giải thích `vrrp_script chk_haproxy`:** Script này kiểm tra định kỳ mỗi 2 giây xem HAProxy có đang chạy không. Nếu HAProxy chết trên node Master, Keepalived sẽ giảm priority và node Backup sẽ giành VIP.

![Cấu hình Keepalived trên cả hai node Master và Backup](https://github.com/user-attachments/assets/f0591c36-8f0a-4cd7-b487-1220867d2675)
*Hình 34: File `keepalived.conf` trên hai node – Node1 có `state MASTER` với `priority 101`, Node2 có `state BACKUP` với `priority 100`. VIP `10.10.10.89` sẽ được gán cho node có priority cao hơn đang hoạt động.*

### Phần 3: Cài Đặt và Cấu Hình HAProxy

**Cài đặt HAProxy trên cả 2 node:**

```bash
sudo apt update
sudo apt install haproxy -y
```

**Cấu hình HAProxy:**

```bash
sudo cp /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg.bak 2>/dev/null || true

sudo cat > /etc/haproxy/haproxy.cfg << 'EOF'
global
    log 127.0.0.1 local2
    chroot /var/lib/haproxy
    pidfile /var/run/haproxy.pid
    maxconn 4000
    user haproxy
    group haproxy
    daemon
    stats socket /var/lib/haproxy/stats

defaults
    mode http
    maxconn 8000
    log global
    option httplog
    option dontlognull
    option http-server-close
    retries 3
    timeout http-request 20s
    timeout queue 1m
    timeout connect 10s
    timeout client 1m
    timeout server 1m
    timeout http-keep-alive 10s
    timeout check 10s

listen stats
    bind *:8080
    mode http
    stats enable
    stats uri /stats
    stats realm HAProxy Statistics
    stats admin if TRUE

listen web-backend
    bind *:80
    balance roundrobin
    cookie SERVERID insert indirect nocache
    mode http
    option httpchk
    option httpclose
    option httplog
    option forwardfor
    server node1 10.10.11.86:8081 check cookie node1 inter 5s fastinter 2s rise 3 fall 3
    server node2 10.10.11.87:8081 check cookie node2 inter 5s fastinter 2s rise 3 fall 3
EOF
```

> **Giải thích cấu hình `listen web-backend`:**
> - `balance roundrobin`: Thuật toán phân phối request lần lượt.
> - `cookie SERVERID insert indirect nocache`: Cấu hình Sticky Sessions qua cookie, giúp client được gắn với cùng một backend server.
> - `check inter 5s fastinter 2s rise 3 fall 3`: Health check mỗi 5 giây; nếu server trả lời liên tiếp 3 lần = `UP`, nếu thất bại 3 lần = `DOWN`.
> - `option forwardfor`: Thêm header `X-Forwarded-For` để backend biết IP thực của client.

Kiểm tra và đảm bảo Apache không còn chiếm cổng 80:

```bash
sudo ss -tulpn | grep ':80'
sudo sed -i "s/<VirtualHost \*:80>/<VirtualHost \*:8081>/g" /etc/apache2/sites-available/000-default.conf
```

![Kết quả kiểm tra cổng và cấu hình Apache đã chuyển sang 8081](https://github.com/user-attachments/assets/d183d322-fead-4a4f-9e20-dd10208b9f62)
*Hình 35: Apache đã được cấu hình lắng nghe tại cổng 8081, không còn xung đột với HAProxy ở cổng 80.*

**Cấu hình Log cho HAProxy:**

```bash
sudo sed -i 's/#module(load="imudp")/module(load="imudp")/' /etc/rsyslog.conf
sudo sed -i 's/#input(type="imudp" port="514")/input(type="imudp" port="514")/' /etc/rsyslog.conf

echo '$UDPServerAddress 127.0.0.1' | sudo tee -a /etc/rsyslog.conf
echo 'local2.*    /var/log/haproxy.log' | sudo tee /etc/rsyslog.d/haproxy.conf

sudo systemctl restart rsyslog
```

