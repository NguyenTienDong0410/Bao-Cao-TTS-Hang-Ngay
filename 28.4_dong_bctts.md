## I. Khái Niệm Cơ Bản về Web Server

### 1. Web Server là gì?

Web Server (máy chủ web) là một phần mềm hoặc hệ thống phần cứng được thiết kế để lưu trữ, xử lý và phân phối các tài nguyên web (trang HTML, hình ảnh, tệp CSS, tệp JavaScript,...) tới người dùng thông qua mạng Internet hoặc mạng nội bộ (Intranet).

Web Server có khả năng tiếp nhận các yêu cầu (request) từ trình duyệt web (browser) của người dùng và gửi phản hồi (response) tương ứng thông qua giao thức HTTP hoặc HTTPS.

> **Ví dụ thực tế:** Khi bạn gõ địa chỉ `https://google.com` vào trình duyệt và nhấn Enter, trình duyệt của bạn đang gửi một yêu cầu HTTP tới Web Server của Google. Web Server nhận yêu cầu, xử lý và trả về trang chủ Google để trình duyệt hiển thị cho bạn. Toàn bộ quá trình đó diễn ra chỉ trong vài trăm mili-giây.

---

### 2. Vai trò của Web Server

Web Server đóng vai trò là cầu nối trung gian giữa máy chủ (server) và máy khách (client), cho phép người dùng truy cập vào các trang web và tài nguyên số thông qua giao thức HTTP/HTTPS.

Cụ thể, Web Server đảm nhận các vai trò sau:

**Xử lý dữ liệu qua giao thức HTTP/HTTPS**

Web Server tiếp nhận các yêu cầu từ trình duyệt của người dùng, xử lý và trả về nội dung tương ứng. Nội dung được phân phối bao gồm: tài liệu HTML, stylesheet (CSS), hình ảnh, tệp JavaScript, video và nhiều định dạng tệp khác.

**Kết nối linh hoạt**

Về mặt lý thuyết, bất kỳ máy tính nào cũng có thể hoạt động như một Web Server nếu được cài đặt phần mềm máy chủ phù hợp (Apache, Nginx,...) và có kết nối Internet. Trong thực tế, các Web Server thường được triển khai trên các máy chủ vật lý chuyên dụng, máy chủ ảo (VPS) hoặc dịch vụ đám mây (Cloud Hosting) để đảm bảo hiệu suất và tính ổn định cao.

**Lưu trữ và phân phối nội dung**

Web Server lưu trữ toàn bộ tệp của một website. Sau khi website được thiết kế và phát triển xong, toàn bộ mã nguồn và tài nguyên phải được tải lên Web Server để người dùng trên toàn thế giới có thể truy cập.

---

### 3. Cách hoạt động của Web Server

Mỗi khi người dùng truy cập một trang web, một chuỗi giao tiếp giữa trình duyệt và Web Server diễn ra theo trình tự sau:

**Bước 1: Trình duyệt phân giải tên miền thành địa chỉ IP**

Trình duyệt không thể kết nối trực tiếp bằng tên miền (ví dụ: `google.com`). Nó cần biết địa chỉ IP thực sự của máy chủ đó. Do đó, trình duyệt gửi yêu cầu tới hệ thống phân giải tên miền DNS (Domain Name System). Máy chủ DNS trả về địa chỉ IP tương ứng (ví dụ: `142.250.196.110`).

> **Ví dụ:** Tên miền `google.com` giống như tên gọi thân thiện, còn địa chỉ IP là địa chỉ nhà thực sự. DNS đóng vai trò như một "danh bạ điện thoại" giúp tra cứu địa chỉ từ tên.

**Bước 2: Trình duyệt gửi yêu cầu HTTP tới Web Server**

Sau khi có địa chỉ IP, trình duyệt thiết lập kết nối TCP đến máy chủ và gửi yêu cầu HTTP chứa thông tin về tài nguyên cần lấy (URL đầy đủ, phiên bản HTTP, thông tin trình duyệt,...).

**Bước 3: Web Server xử lý và phản hồi**

Web Server nhận yêu cầu, tìm kiếm tài nguyên được yêu cầu trong hệ thống tệp hoặc thông qua ứng dụng phía máy chủ, sau đó gửi lại phản hồi. Phản hồi bao gồm mã trạng thái HTTP (200 OK, 404 Not Found, 500 Internal Server Error,...) và nội dung tài nguyên tương ứng.

**Bước 4: Trình duyệt hiển thị trang web**

Trình duyệt nhận các tệp HTML, CSS, JavaScript từ phản hồi và tiến hành dựng (render) giao diện trang web để hiển thị cho người dùng.

## II. Các Loại Web Server Phổ Biến

### 1. Apache HTTP Server

Apache HTTP Server là một trong những Web Server mã nguồn mở lâu đời và phổ biến nhất thế giới. Được phát triển bởi Apache Software Foundation, Apache sử dụng giao thức HTTP để phục vụ các trang web và ứng dụng web. Web Server này thường được kết hợp với các ngôn ngữ lập trình như PHP, Python hoặc Ruby để xây dựng ứng dụng web động, và nổi tiếng nhờ tính ổn định, khả năng mở rộng cao cùng hệ thống module linh hoạt.

Apache hoạt động theo mô hình xử lý đồng bộ đa tiến trình (Multi-Processing Module - MPM), trong đó mỗi yêu cầu HTTP sẽ được xử lý bởi một tiến trình hoặc luồng riêng biệt.

**Ưu điểm của Apache**

- **Miễn phí và mã nguồn mở:** Apache không phát sinh chi phí bản quyền và cho phép tùy chỉnh mã nguồn theo nhu cầu.
- **Phổ biến và ổn định:** Được sử dụng rộng rãi trong nhiều thập kỷ, Apache đã được kiểm nghiệm thực tế trên quy mô lớn.
- **Đa nền tảng:** Chạy được trên Linux, Windows, macOS và nhiều hệ điều hành khác.
- **Hệ thống module linh hoạt:** Apache hỗ trợ hàng trăm module bổ sung (mod_rewrite, mod_ssl, mod_php,...) có thể bật hoặc tắt tùy theo nhu cầu.
- **Bảo mật:** Có cộng đồng lớn thường xuyên theo dõi, phát hiện và vá lỗi bảo mật.
- **Hỗ trợ nhiều ngôn ngữ:** Tương thích với PHP, Python, Ruby, Perl và nhiều ngôn ngữ lập trình khác.
- **Tài liệu và cộng đồng phong phú:** Dễ dàng tìm kiếm hướng dẫn, tài liệu tham khảo và hỗ trợ kỹ thuật từ cộng đồng.

**Nhược điểm của Apache**

- **Tiêu tốn nhiều tài nguyên:** Mô hình xử lý đồng bộ khiến Apache tốn nhiều RAM và CPU khi có nhiều kết nối đồng thời. Mỗi kết nối mới đều tạo ra một tiến trình/luồng riêng.
- **Khả năng mở rộng có giới hạn:** Khi số lượng kết nối đồng thời vượt quá hàng nghìn, hiệu suất sẽ giảm rõ rệt nếu không tinh chỉnh cấu hình.
- **Không hỗ trợ HTTP/2 theo mặc định:** Phiên bản cơ bản chỉ hỗ trợ HTTP/1.1; cần cấu hình thêm để sử dụng HTTP/2.
- **Cấu hình phức tạp:** Cú pháp tệp cấu hình (httpd.conf, .htaccess) tương đối phức tạp và dễ gây nhầm lẫn với người mới bắt đầu.
- **Hiệu năng bị ảnh hưởng bởi module bên thứ ba:** Việc kết hợp nhiều module có thể ảnh hưởng đến hiệu năng và tính ổn định tổng thể.

### 2. Nginx

Nginx (đọc là "Engine-X") là một phần mềm mã nguồn mở được sử dụng làm Web Server, Reverse Proxy, Load Balancer và HTTP Cache. Nginx nổi bật nhờ khả năng xử lý hàng chục nghìn kết nối đồng thời với mức tiêu thụ tài nguyên rất thấp.

Nginx được phát triển bởi Igor Sysoev vào năm 2002, ban đầu nhằm phục vụ cho Rambler.ru - một trong những trang web có lưu lượng truy cập lớn nhất nước Nga. Nginx được phát hành dưới giấy phép BSD và viết bằng ngôn ngữ C, hỗ trợ chạy trên Linux, FreeBSD, Windows, macOS.

![Kiến trúc và mô hình hoạt động của Nginx](https://github.com/user-attachments/assets/d8279a69-768d-4b19-b22c-b8355b5ddd30)

**Kiến trúc bất đồng bộ (Asynchronous / Event-driven)**

Điểm khác biệt cốt lõi của Nginx so với Apache là kiến trúc xử lý yêu cầu:

- **Apache (đa luồng - Multi-threaded):** Mỗi khi có một yêu cầu từ người dùng, Apache tạo ra một tiến trình hoặc luồng riêng để xử lý yêu cầu đó. Với 1.000 kết nối đồng thời, Apache cần tạo và duy trì 1.000 luồng, tiêu tốn rất nhiều bộ nhớ RAM.
- **Nginx (bất đồng bộ - Event Loop):** Nginx sử dụng cơ chế vòng lặp sự kiện (event loop). Một luồng chính duy nhất lắng nghe và phản hồi tất cả các sự kiện đến mà không cần tạo luồng mới cho từng yêu cầu. Với 1.000 kết nối đồng thời, Nginx vẫn chỉ cần một luồng duy nhất xử lý, tiết kiệm tài nguyên đáng kể.

> **Ví dụ hình ảnh:** Hãy tưởng tượng Apache như một nhà hàng với mô hình "mỗi khách hàng có một phục vụ viên riêng" - khi có 100 khách, cần 100 phục vụ viên. Nginx giống một phục vụ viên duy nhất có thể phục vụ 100 bàn cùng lúc bằng cách lắng nghe và xử lý tuần tự các yêu cầu ngay khi chúng sẵn sàng.

**Ưu điểm của Nginx**

- **Hiệu năng cao với lưu lượng lớn:** Nhờ kiến trúc bất đồng bộ, Nginx xử lý tốt vấn đề C10K (10.000 kết nối đồng thời) mà không gây tắc nghẽn.
- **Tốc độ tải trang nhanh:** Nginx phân phối nội dung tĩnh (ảnh, CSS, JS) cực kỳ hiệu quả, giảm thời gian phản hồi cho người dùng.
- **Reverse Proxy và Load Balancing:** Nginx có thể đứng trước nhiều máy chủ ứng dụng, phân phối đều lưu lượng truy cập mà không cần thiết bị phần cứng load balancer vật lý.
- **Hệ thống module linh hoạt:** Các module như SSL, nén dữ liệu (gzip), bộ nhớ đệm (cache) có thể được tích hợp theo nhu cầu.
- **Hỗ trợ nhiều giao thức:** HTTP/1.x, HTTP/2, HTTPS, WebSocket, IMAP, POP3, SMTP, TCP, UDP.
- **SSL Termination:** Nginx có khả năng xử lý mã hóa SSL/TLS tại tầng proxy và chuyển tiếp dữ liệu không mã hóa tới các máy chủ nội bộ, giảm tải cho máy chủ ứng dụng.

**Nhược điểm của Nginx**

- **Không hỗ trợ xử lý động trực tiếp:** Nginx không tích hợp sẵn bộ biên dịch ngôn ngữ động như PHP. Để chạy PHP, cần kết hợp với PHP-FPM qua giao thức FastCGI.
- **Một số tính năng nâng cao chỉ có trong bản thương mại (Nginx Plus):** Ví dụ như Active Health Checks, dashboard giám sát chi tiết.
- **Cú pháp cấu hình khác biệt:** Người dùng quen với Apache cần thời gian để làm quen với cú pháp cấu hình của Nginx.

### 3. Internet Information Services (IIS)

IIS (Internet Information Services) là một Web Server do Microsoft phát triển, chạy trên hệ điều hành Windows và được tích hợp sẵn trong các phiên bản Windows Server. IIS hỗ trợ các giao thức HTTP, HTTPS, FTP, SMTP và NNTP, giúp các doanh nghiệp dễ dàng quản lý và phân phối nội dung web trên môi trường Internet lẫn Intranet.

**Lịch sử phát triển**

IIS đã trải qua nhiều phiên bản với những cải tiến liên tục về tính năng và hiệu suất:

| Phiên bản | Nền tảng | Điểm nổi bật |
| :--- | :--- | :--- |
| IIS 1.0 | Windows NT 3.51 | Phiên bản đầu tiên |
| IIS 4.0 | Windows NT 4.0 | Cải tiến giao diện quản lý |
| IIS 5.0 | Windows 2000 | Bổ sung tính năng bảo mật mới |
| IIS 6.0 | Windows Server 2003 | Nâng cấp đáng kể về hiệu suất và khả năng mở rộng |
| IIS 7.0 | Windows Server 2008 / Vista | Giới thiệu kiến trúc module, tùy chỉnh linh hoạt hơn |
| IIS 8.0 | Windows Server 2012 | Cải tiến hiệu năng và bảo mật |
| IIS 10.0 | Windows Server 2016 / Windows 10 | Hỗ trợ HTTP/2, container hóa |

**Cơ chế hoạt động của IIS**

IIS hoạt động theo kiến trúc hai lớp:

- **Kernel Mode:** Tiếp nhận và phân tích các yêu cầu HTTP đến, xác định ứng dụng phù hợp để xử lý. Lớp này tận dụng tính năng HTTP.sys của hệ điều hành Windows để xử lý giao thức HTTP ở mức nhân, giúp tăng tốc độ và bảo mật.
- **User Mode:** Nhận yêu cầu đã được xử lý từ Kernel Mode, thực thi ứng dụng web và trả về kết quả. Các ứng dụng web chạy trong Application Pool, được cô lập với nhau để một ứng dụng lỗi không ảnh hưởng đến các ứng dụng khác.

![Mô hình xử lý hai lớp của IIS](https://github.com/user-attachments/assets/c75b2760-76bb-47af-bdec-79973c4e47d7)

**Tính năng nổi bật của IIS**

- **Hỗ trợ ASP.NET và nhiều ngôn ngữ:** IIS là nền tảng lý tưởng cho ASP.NET nhưng cũng hỗ trợ PHP, Perl và nhiều ngôn ngữ khác.
- **Bảo mật FTP:** Cung cấp xác thực, mã hóa và kiểm soát quyền truy cập cho máy chủ FTP.
- **Nhiều phương thức xác thực:** Hỗ trợ xác thực cơ bản, xác thực Windows (Kerberos/NTLM) và xác thực bằng chứng chỉ số.
- **Mã hóa SSL/TLS:** Bảo vệ dữ liệu truyền giữa trình duyệt và máy chủ.
- **Kiến trúc module:** Cho phép bật/tắt các tính năng theo nhu cầu, giảm bề mặt tấn công.
- **Application Pool:** Cô lập các ứng dụng web trong các tiến trình riêng biệt, tăng tính ổn định.

**Ứng dụng phổ biến**

IIS thường được sử dụng để lưu trữ các trang web tĩnh của doanh nghiệp, các ứng dụng thương mại điện tử hoặc mạng xã hội, và các ứng dụng doanh nghiệp nội bộ như hệ thống CRM (Customer Relationship Management) hay ERP (Enterprise Resource Planning).

### 4. Lighttpd

Lighttpd (đọc là "lighty") là một phần mềm Web Server mã nguồn mở, nhẹ, nhanh và tiết kiệm tài nguyên, được viết bởi Jan Kneschke bằng ngôn ngữ C. Lighttpd được phát hành theo giấy phép BSD và chạy được trên Linux, Windows, macOS.

Điểm mạnh nổi bật nhất của Lighttpd là mức tiêu thụ CPU và RAM cực kỳ thấp, khiến nó phù hợp với các môi trường hạn chế tài nguyên như thiết bị nhúng (embedded systems) hay các máy chủ nhỏ.

Lighttpd hỗ trợ FastCGI, CGI và SCGI, cho phép chạy ứng dụng viết bằng PHP, Python, Perl, Ruby, Lua và nhiều ngôn ngữ khác. Tương tự Nginx, Lighttpd xử lý các yêu cầu theo mô hình bất đồng bộ, có thể phục vụ nhiều kết nối cùng lúc mà không làm chậm hệ thống.

![So sánh hiệu năng Lighttpd](https://github.com/user-attachments/assets/c9e9782a-0b41-4cbb-9aa8-78e50d0f7786)

**Ưu điểm của Lighttpd**

- Tiêu thụ tối thiểu CPU, RAM và các tài nguyên khác.
- Hỗ trợ hầu hết các ngôn ngữ lập trình qua FastCGI/CGI.
- Hoàn toàn miễn phí và mã nguồn mở.
- Phù hợp cho các thiết bị nhúng, router, hoặc các server nhỏ.

**Nhược điểm của Lighttpd**

- Không phổ biến bằng Apache hay Nginx, vì vậy tài liệu và hỗ trợ cộng đồng còn hạn chế.
- Thiếu các tính năng nâng cao và hệ sinh thái module phong phú như hai đối thủ trên.
- Hiệu năng bị hạn chế với các trang web lớn, lưu lượng truy cập cao.

> **Khi nào nên dùng Lighttpd?** Lighttpd phù hợp nhất khi bạn cần một Web Server nhẹ, đơn giản cho các ứng dụng quy mô nhỏ hoặc môi trường phần cứng hạn chế. Không nên dùng Lighttpd cho các hệ thống sản xuất lớn cần tính năng cao cấp.

### 5. Caddy

![Caddy Server](https://github.com/user-attachments/assets/347f9343-76c4-40c1-a601-96c8bace455e)

Caddy Server là một Web Server mã nguồn mở, viết bằng ngôn ngữ Go, được phát triển với triết lý "đơn giản là tốt nhất". Caddy hướng đến mục tiêu cấu hình đơn giản, hiệu năng cao và mặc định sử dụng HTTPS cho tất cả các trang web.

Caddy có đầy đủ các tính năng cốt lõi của Nginx như gzip, chỉnh sửa header, xác thực, load balancing và thậm chí có thể tự động render tệp Markdown thành HTML.

**Tính năng chính của Caddy Server**

- **HTTPS tự động:** Caddy tự động xin và gia hạn chứng chỉ TLS từ Let's Encrypt cho tất cả tên miền được cấu hình. Không cần cấu hình SSL thủ công như Apache hay Nginx - đây là tính năng nổi bật nhất của Caddy.
- **Cấu hình đơn giản:** File cấu hình Caddyfile có cú pháp rất dễ đọc và dễ viết, phù hợp cả với người mới bắt đầu.
- **Khả năng mở rộng qua Plugin:** Hỗ trợ hệ sinh thái plugin để mở rộng chức năng theo nhu cầu.
- **API quản trị (Admin API):** Cho phép thay đổi cấu hình trực tiếp qua REST API mà không cần khởi động lại dịch vụ.
- **TLS theo yêu cầu (On-Demand TLS):** Caddy có thể cấp chứng chỉ TLS ngay trong quá trình kết nối, rất hữu ích trong các môi trường đa tên miền động.

**Quy trình sử dụng Caddy Server**

1. Kích hoạt Daemon bằng lệnh `caddy run`.
2. Sử dụng Admin API để kiểm tra và điều chỉnh cấu hình.
3. Cung cấp cấu hình qua Caddyfile hoặc JSON.
4. Xác minh hoạt động bằng cách truy cập `localhost:2015` hoặc dùng `curl`.
5. Có thể dùng bộ chuyển đổi (adapter) để chuyển Caddyfile sang định dạng JSON gốc mà Caddy sử dụng nội bộ.

**Ưu điểm của Caddy Server**

- HTTPS tự động, không cần cấu hình chứng chỉ thủ công.
- Cú pháp cấu hình đơn giản, thân thiện với người dùng.
- Viết bằng Go - biên dịch thành tệp nhị phân đơn lẻ, không phụ thuộc thư viện ngoài.
- Cập nhật cấu hình không cần khởi động lại dịch vụ.

**Nhược điểm của Caddy Server**

- Còn tương đối mới so với Apache và Nginx, thiếu kinh nghiệm vận hành thực chiến.
- Hệ sinh thái module và plugin còn hạn chế.
- Cộng đồng hỗ trợ nhỏ hơn đáng kể.
- Hiệu suất bị giới hạn với các trang web có lưu lượng cực lớn.

- **Tính "hộp đen" (Black Box):** Không thể kiểm tra hoặc kiểm toán mã nguồn, hệ thống phụ thuộc hoàn toàn vào năng lực giải quyết lỗi của nhà phát hành.

## IV. Giao Thức và Công Nghệ Liên Quan

### 1. HTTP và HTTPS

#### 1.1 HTTP là gì?

HTTP (Hypertext Transfer Protocol - Giao thức Truyền tải Siêu văn bản) là giao thức nền tảng của World Wide Web, được sử dụng để truyền tải dữ liệu (văn bản, hình ảnh, âm thanh, video,...) giữa Web Server và trình duyệt của người dùng.

HTTP hoạt động theo mô hình Client - Server:

1. Client (trình duyệt) gửi yêu cầu (HTTP Request) đến Server.
2. Server xử lý yêu cầu và trả về phản hồi (HTTP Response).
3. Trình duyệt hiển thị nội dung nhận được.

**Hạn chế của HTTP:** Toàn bộ dữ liệu truyền qua HTTP đều ở dạng văn bản thuần túy (plain text), không được mã hóa. Điều này khiến HTTP dễ bị tấn công kiểu "nghe lén" (sniffing attack) - hacker có thể đọc được mọi thông tin bạn gửi đi, bao gồm mật khẩu, thông tin thẻ ngân hàng,...

#### 1.2 HTTPS là gì?

HTTPS (Hypertext Transfer Protocol Secure) là phiên bản bảo mật của HTTP, tích hợp thêm lớp mã hóa SSL/TLS để bảo vệ dữ liệu trong quá trình truyền tải. HTTPS là tiêu chuẩn bảo mật tối thiểu mà mọi website hiện đại cần phải đáp ứng.

HTTPS bổ sung thêm chứng chỉ SSL (Secure Sockets Layer) hoặc TLS (Transport Layer Security) vào giao thức HTTP. Cả SSL và TLS đều sử dụng hệ thống PKI (Public Key Infrastructure - hạ tầng khóa công khai) không đối xứng với hai loại khóa:

- **Khóa công khai (Public Key):** Được chia sẻ công khai, dùng để mã hóa dữ liệu gửi đến.
- **Khóa riêng (Private Key):** Chỉ máy chủ mới có, dùng để giải mã dữ liệu nhận được.

#### 1.3 So sánh HTTP và HTTPS

| Tiêu chí | HTTP | HTTPS |
| :--- | :--- | :--- |
| **Bảo mật** | Không mã hóa, dễ bị nghe lén | Mã hóa SSL/TLS, an toàn |
| **Cổng mặc định** | Port 80 | Port 443 |
| **Chứng chỉ** | Không yêu cầu | Yêu cầu chứng chỉ SSL/TLS từ CA |
| **Tốc độ** | Nhanh hơn (không có overhead mã hóa) | Chậm hơn một chút, nhưng HTTP/2 bù đắp phần lớn |
| **Biểu tượng trình duyệt** | Cảnh báo "Không bảo mật" | Biểu tượng ổ khóa |
| **SEO** | Không được ưu tiên | Google ưu tiên xếp hạng cao hơn |

**Tại sao nên sử dụng HTTPS?**

- **Bảo mật thông tin người dùng:** Mã hóa ngăn chặn hacker đọc dữ liệu truyền đi (mật khẩu, thông tin cá nhân, thẻ tín dụng).
- **Chống giả mạo (Anti-phishing):** Chứng chỉ SSL xác minh danh tính của website, đảm bảo người dùng đang kết nối đúng địa chỉ chứ không phải website giả mạo.
- **Tăng uy tín:** Các trình duyệt như Chrome, Firefox, Edge đều cảnh báo người dùng khi truy cập website HTTP, làm giảm niềm tin của khách hàng.
- **Cải thiện SEO:** Google và các công cụ tìm kiếm ưu tiên xếp hạng cao hơn cho các website sử dụng HTTPS.

### 2. SSL/TLS

#### 2.1 SSL là gì?

SSL (Secure Sockets Layer) là giao thức bảo mật được Netscape phát triển lần đầu vào năm 1995, nhằm thiết lập một kết nối mã hóa giữa máy chủ và máy khách, đảm bảo tính riêng tư, xác thực và toàn vẹn của dữ liệu. Tuy nhiên, SSL 2.0 và 3.0 dần bộc lộ các lỗ hổng bảo mật nghiêm trọng theo thời gian.

#### 2.2 TLS là gì?

TLS (Transport Layer Security) là phiên bản kế thừa và cải tiến của SSL, được IETF (Internet Engineering Task Force) đề xuất vào năm 1999. TLS khắc phục các điểm yếu của SSL và bổ sung nhiều tính năng bảo mật tiên tiến hơn. Ngày nay, TLS đã thay thế hoàn toàn SSL, tuy nhiên thuật ngữ "SSL" vẫn được nhiều người sử dụng để chỉ cả hai.

> **Lưu ý:** Biểu tượng ổ khóa trên thanh địa chỉ của trình duyệt là dấu hiệu cho thấy website đang sử dụng TLS để bảo vệ kết nối.

#### 2.3 Các phương thức mã hóa SSL/TLS sử dụng

SSL/TLS kết hợp hai phương thức mã hóa để cân bằng giữa tốc độ và bảo mật:

**Mã hóa đối xứng (Symmetric Cryptography)**

Sử dụng một khóa duy nhất cho cả mã hóa và giải mã.

```
Dữ liệu + Khóa bí mật  →  Thuật toán mã hóa  →  Dữ liệu đã mã hóa
Dữ liệu đã mã hóa + Khóa bí mật  →  Giải mã  →  Dữ liệu gốc
```

- **Ưu điểm:** Tốc độ nhanh, phù hợp xử lý lượng lớn dữ liệu.
- **Nhược điểm:** Cần trao đổi khóa bí mật qua mạng - dễ bị đánh cắp nếu không có cơ chế bảo vệ.

**Mã hóa bất đối xứng (Asymmetric Cryptography)**

Sử dụng cặp khóa: Public Key (khóa công khai) và Private Key (khóa riêng).

- Dữ liệu mã hóa bằng Public Key chỉ có thể giải mã bằng Private Key tương ứng.
- Public Key được chia sẻ công khai; Private Key chỉ máy chủ mới giữ.

> **Ví dụ:** Client muốn gửi thông tin "1234" cho Server. Client dùng Public Key của Server để mã hóa thành "!@#$" và gửi đi. Chỉ Server có Private Key mới có thể giải mã và lấy lại "1234". Dù hacker chặn được gói tin "!@#$", họ cũng không thể giải mã mà không có Private Key.

- **Ưu điểm:** Trao đổi khóa an toàn.
- **Nhược điểm:** Tốc độ chậm hơn đáng kể so với mã hóa đối xứng.

**Tại sao SSL kết hợp cả hai?**

SSL sử dụng mã hóa bất đối xứng trong giai đoạn bắt tay (handshake) để trao đổi khóa bí mật một cách an toàn, sau đó dùng khóa bí mật đó cho mã hóa đối xứng trong suốt phiên truyền dữ liệu. Cách này tận dụng được điểm mạnh của cả hai: bảo mật của mã hóa bất đối xứng và tốc độ của mã hóa đối xứng.

#### 2.4 Chứng chỉ SSL là gì?

Chứng chỉ SSL là một tệp dữ liệu số do tổ chức bên thứ ba gọi là Certificate Authority (CA - Cơ quan chứng nhận) cấp. Chứng chỉ này chứa thông tin về chủ sở hữu máy chủ, khóa công khai và chữ ký số của CA, xác nhận rằng máy chủ đó là chính thống.

Khi truy cập một website HTTPS, trình duyệt tự động kiểm tra chứng chỉ SSL. Nếu hợp lệ, biểu tượng ổ khóa xuất hiện trên thanh địa chỉ.

#### 2.5 Quá trình SSL Handshake

SSL Handshake là quá trình hai bên (client và server) thỏa thuận các tham số mã hóa trước khi truyền dữ liệu thực sự. Quá trình diễn ra như sau:

1. **Client Hello:** Client gửi phiên bản SSL/TLS hỗ trợ, danh sách cipher suites (bộ thuật toán mã hóa) và một chuỗi ngẫu nhiên (Client Random).
2. **Server Hello:** Server chọn phiên bản SSL/TLS và cipher suite phù hợp, gửi lại Server Random cùng với chứng chỉ SSL của mình.
3. **Xác thực chứng chỉ:** Client kiểm tra xem chứng chỉ có hợp lệ và được ký bởi CA đáng tin cậy không.
4. **Trao đổi Premaster Secret:** Client tạo một chuỗi byte bí mật (Premaster Secret), mã hóa bằng Public Key của Server, rồi gửi đi.
5. **Giải mã Premaster Secret:** Server dùng Private Key để giải mã và lấy lại Premaster Secret.
6. **Tạo Session Key:** Cả hai bên dùng Client Random + Server Random + Premaster Secret để tạo ra một Session Key dùng cho mã hóa đối xứng trong phiên này.
7. **Hoàn tất Handshake:** Cả hai bên xác nhận và bắt đầu truyền dữ liệu được mã hóa bằng Session Key.

### 3. WebSocket, HTTP/2 và QUIC

#### 3.1 WebSocket

WebSocket là một giao thức truyền thông cung cấp kênh liên lạc song công hoàn toàn (full-duplex) qua một kết nối TCP duy nhất. Khác với HTTP truyền thống theo mô hình yêu cầu-phản hồi (client phải gửi request mới nhận được dữ liệu từ server), WebSocket cho phép cả hai phía chủ động gửi dữ liệu cho nhau bất cứ lúc nào.

**Cách hoạt động của WebSocket**

1. Client gửi một HTTP Request đặc biệt (chứa header `Upgrade: websocket`) đến Server, yêu cầu nâng cấp kết nối từ HTTP sang WebSocket.
2. Server phản hồi với mã trạng thái `101 Switching Protocols`, xác nhận chuyển đổi giao thức.
3. Kết nối WebSocket được thiết lập. Từ đây, cả client và server đều có thể gửi dữ liệu cho nhau bất cứ lúc nào mà không cần gửi request trước.
4. Dữ liệu được đóng gói trong các "khung" (WebSocket Frame), có thể chứa văn bản hoặc dữ liệu nhị phân.
5. Kết nối đóng khi một trong hai bên chủ động đóng hoặc mạng bị gián đoạn.

> **Ví dụ dễ hình dung:** HTTP giống như giao tiếp qua thư - bạn gửi thư (request) và chờ nhận thư trả lời (response). WebSocket giống như gọi điện thoại - khi đã kết nối, cả hai bên có thể nói chuyện bất cứ lúc nào mà không cần "xin phép" trước.

**Ứng dụng phổ biến của WebSocket**

- **Chat trực tuyến:** Tin nhắn được truyền tức thời giữa các người dùng mà không cần tải lại trang.
- **Trò chơi trực tuyến (Online Gaming):** Đồng bộ hóa trạng thái trò chơi giữa các người chơi theo thời gian thực.
- **Theo dõi thị trường tài chính:** Cập nhật giá cổ phiếu, tỷ giá ngoại tệ liên tục.
- **Truyền phát video và âm thanh (Streaming):** Đảm bảo luồng dữ liệu liên tục, không bị gián đoạn.
- **Internet of Things (IoT):** Kết nối liên tục giữa các thiết bị thông minh trong hệ thống.

**Hạn chế của WebSocket**

- Một số trình duyệt và proxy cũ không hỗ trợ hoặc hỗ trợ không đầy đủ.
- Duy trì kết nối liên tục tiêu tốn tài nguyên máy chủ khi có nhiều người dùng đồng thời.
- Không phải là lựa chọn tối ưu trong môi trường mạng có băng thông hẹp hoặc độ trễ cao.
- Triển khai và debug phức tạp hơn HTTP thông thường.

**So sánh HTTP và WebSocket**

| Tiêu chí | HTTP | WebSocket |
| :--- | :--- | :--- |
| **Loại kết nối** | Yêu cầu - Phản hồi (Request-Response) | Hai chiều, liên tục (Full-Duplex) |
| **Duy trì trạng thái** | Không (Stateless) | Có (Stateful) |
| **Ứng dụng** | Trang web tĩnh, ứng dụng không yêu cầu truyền thông liên tục | Ứng dụng thời gian thực, chat, game |
| **Tối ưu hóa tài nguyên** | Phù hợp cho các yêu cầu đơn lẻ | Giảm tải mạng với kết nối duy trì |
| **Tương thích** | Rộng rãi trên tất cả trình duyệt và máy chủ | Yêu cầu hỗ trợ từ trình duyệt và máy chủ |

#### 3.2 HTTP/2

HTTP/2 là phiên bản kế tiếp của HTTP, được công bố chính thức vào năm 2015 qua RFC 7540. HTTP/2 được phát triển dựa trên nền tảng giao thức SPDY của Google, với mục tiêu cải thiện đáng kể hiệu suất tải trang so với HTTP/1.1.

![HTTP/2 Stack](https://github.com/user-attachments/assets/82aace7c-869d-4d3a-88c1-5b9ee55335b3)

**Các tính năng nổi bật của HTTP/2**

**Binary Protocol (Giao thức nhị phân)**

HTTP/1.1 truyền dữ liệu dưới dạng văn bản thuần túy (text), dễ đọc nhưng khó phân tích tự động và dễ mắc lỗi. HTTP/2 chuyển sang định dạng nhị phân (binary), giúp quá trình phân tích nhanh hơn, gọn hơn và ít lỗi hơn.

**Request-Response Multiplexing (Đa hợp)**

Trong HTTP/1.1, dù có cơ chế Pipelining (gửi nhiều request liên tiếp), phía client vẫn phải nhận response theo đúng thứ tự đã gửi. Điều này gây ra hiện tượng Head-of-Line Blocking (HOL) - một response chậm sẽ chặn toàn bộ pipeline.

HTTP/2 giải quyết vấn đề này bằng Multiplexing: mỗi request/response được định danh độc lập (stream ID), có thể truyền xen kẽ nhau trên cùng một kết nối TCP mà không cần chờ theo thứ tự.

> **Ví dụ:** Thay vì tải lần lượt 10 ảnh nhỏ (ảnh 1 xong mới tải ảnh 2,...), HTTP/2 chia nhỏ dữ liệu và tải đồng thời tất cả 10 ảnh trên cùng một kết nối.

**Streams và Stream Prioritization**

Kết nối HTTP/2 bao gồm nhiều Stream (luồng dữ liệu), mỗi stream chứa các Data Frame. HTTP/2 cho phép client chỉ định mức độ ưu tiên cho từng stream, giúp server tối ưu hóa phân bổ tài nguyên (ví dụ: tải CSS trước để trang hiển thị nhanh hơn).

**Stateful Header Compression (HPACK)**

HTTP/1.1 gửi toàn bộ header trong mỗi request, kể cả các header lặp đi lặp lại (User-Agent, Cookie,...). HTTP/2 sử dụng thuật toán nén HPACK, lưu trữ các header đã sử dụng và chỉ truyền phần thay đổi, giảm đáng kể lượng dữ liệu header.

**Server Push**

HTTP/2 cho phép server chủ động gửi tài nguyên tới client mà không cần client phải request trước. Ví dụ: khi client request tệp HTML, server có thể ngay lập tức gửi kèm tệp CSS và JavaScript mà nó biết client sẽ cần, tiết kiệm các round-trip không cần thiết.

#### 3.3 QUIC

QUIC (Quick UDP Internet Connections) là một giao thức truyền tải mạng thế hệ mới, được Google phát triển từ năm 2012 và sau đó được IETF chuẩn hóa. QUIC sử dụng UDP thay vì TCP như các giao thức truyền thống, nhằm cải thiện tốc độ, giảm độ trễ và tăng bảo mật.

> **Lưu ý:** Tồn tại hai phiên bản QUIC: Google QUIC (gQUIC) là phiên bản gốc của Google, và IETF QUIC (thường gọi tắt là QUIC) là phiên bản được chuẩn hóa từ năm 2016, có nhiều cải tiến đáng kể so với bản gốc. QUIC phiên bản IETF được xem là một giao thức hoàn toàn mới.

Có thể hiểu QUIC là sự kết hợp của TCP + TLS + HTTP/2 nhưng được xây dựng trên nền UDP. Lý do không cải tiến trực tiếp TCP là vì TCP đã được tích hợp sâu vào nhân hệ điều hành (OS Kernel) và các thiết bị mạng trung gian (firewall, router,...) trong nhiều thập kỷ, khiến việc thay đổi gần như không thể. QUIC không bị giới hạn này vì được xây dựng trên UDP ở tầng ứng dụng.

![So sánh stack giao thức](https://github.com/user-attachments/assets/796d0692-4d94-4669-b57e-5304011b49b5)

**Cách hoạt động của QUIC**

- **Handshake nhanh hơn:** QUIC kết hợp quá trình thiết lập kết nối và TLS Handshake vào một bước, giảm số round-trip cần thiết (HTTP/2 qua TCP cần 2-3 round-trip; QUIC thường chỉ cần 1, thậm chí 0 với kết nối đã biết trước).
- **Mã hóa mặc định:** QUIC sử dụng TLS 1.3 để mã hóa toàn bộ dữ liệu, bao gồm cả header giao thức.
- **Kiểm soát lưu lượng thông minh:** QUIC tự điều chỉnh tốc độ truyền tải phù hợp với điều kiện mạng thực tế.
- **Khắc phục Head-of-Line Blocking ở mức transport:** Khác với TCP (một gói tin bị mất sẽ chặn toàn bộ kết nối), QUIC xử lý từng stream độc lập - một gói tin bị mất chỉ ảnh hưởng đến stream đó, không chặn các stream khác.

**Ưu điểm nổi bật của QUIC**

- **Tốc độ kết nối nhanh:** Giảm thiểu thời gian thiết lập kết nối và độ trễ ban đầu.
- **Bảo mật tích hợp:** TLS 1.3 được nhúng trực tiếp vào giao thức, không thể bỏ qua.
- **Khả năng chuyển mạng (Connection Migration):** Khi người dùng chuyển từ WiFi sang 4G, kết nối QUIC vẫn duy trì được mà không cần thiết lập lại, nhờ sử dụng Connection ID thay vì bộ đôi (IP, Port) như TCP.
- **Hỗ trợ đa kết nối song song:** Tận dụng tối đa băng thông mạng.

**QUIC và HTTP/3**

QUIC là nền tảng vận chuyển của HTTP/3 - phiên bản HTTP mới nhất. HTTP/3 thay thế TCP bằng QUIC, mang lại hiệu suất truyền tải tốt hơn đáng kể, đặc biệt trong các môi trường mạng kém ổn định như mạng di động.

### 4. CGI, FastCGI và WSGI

#### 4.1 Tổng quan: Tại sao cần các giao thức này?

Các Web Server (Apache, Nginx,...) được tối ưu để phân phối nội dung tĩnh (HTML, CSS, ảnh). Tuy nhiên, chúng không thể tự biên dịch và thực thi mã logic của các ngôn ngữ lập trình như PHP, Python, Ruby,... Do đó cần có một "cầu nối" tiêu chuẩn giữa Web Server và ứng dụng động. CGI, FastCGI và WSGI chính là các tiêu chuẩn giao thức đóng vai trò trung gian này.

---

#### 4.2 CGI (Common Gateway Interface)

**Định nghĩa**

CGI (Common Gateway Interface - Giao diện Cổng Chung) là tiêu chuẩn giao tiếp đầu tiên giữa Web Server và các chương trình ứng dụng bên ngoài. Theo chuẩn CGI, Web Server sẽ gọi một chương trình (CGI script) để xử lý request và trả về kết quả.

**Cơ chế hoạt động**

1. Client gửi yêu cầu đến Web Server.
2. Web Server nhận yêu cầu và khởi động một tiến trình mới để thực thi CGI script tương ứng.
3. Web Server truyền thông tin về yêu cầu (URL, tham số, dữ liệu POST,...) cho CGI script thông qua biến môi trường (environment variables) và standard input.
4. CGI script xử lý, tạo ra nội dung phản hồi và ghi ra standard output.
5. Web Server đọc output của CGI script và trả về cho client.
6. Tiến trình CGI bị hủy sau khi hoàn thành.

> **Ví dụ:** Giống như mỗi khi cần một tờ giấy, bạn phải thuê một người đến làm việc đó, xong việc thì họ về. Nếu có 1.000 người cùng cần giấy một lúc, bạn phải thuê 1.000 người - rất tốn kém và không thực tế.

**Ưu điểm**

- Kiến trúc đơn giản, dễ hiểu.
- Độc lập với ngôn ngữ lập trình (có thể viết CGI bằng bất kỳ ngôn ngữ nào).
- Tính cô lập cao: mỗi yêu cầu chạy trong tiến trình riêng, lỗi không lan sang request khác.

**Nhược điểm nghiêm trọng**

- **Hiệu suất cực kỳ thấp:** Mỗi request tạo ra một tiến trình mới (fork overhead). Khởi tạo và hủy tiến trình liên tục tiêu tốn rất nhiều CPU và RAM.
- **Không lưu trạng thái:** Bộ nhớ, kết nối cơ sở dữ liệu không được tái sử dụng - phải khởi tạo lại từ đầu với mỗi request.
- **Dễ quá tải:** Với lưu lượng cao, hệ thống sẽ cạn kiệt tài nguyên nhanh chóng.

CGI thuần túy hiện nay gần như không còn được sử dụng trong các hệ thống sản xuất thực tế.

---

#### 4.3 FastCGI (Fast Common Gateway Interface)

**Định nghĩa và bối cảnh ra đời**

FastCGI là một giao thức nhị phân (binary protocol) ra đời để khắc phục điểm yếu cốt lõi của CGI: chi phí khởi tạo tiến trình. Thay vì tạo và hủy tiến trình cho mỗi request, FastCGI sử dụng cơ chế tiến trình thường trú (persistent processes) - các tiến trình được khởi động một lần và xử lý nhiều request liên tiếp mà không cần khởi động lại.

**Cơ chế hoạt động**

1. **Khởi tạo:** Khi hệ thống khởi động, FastCGI Process Manager tạo ra một nhóm (pool) các tiến trình worker sẵn sàng chờ lệnh.
2. **Tiếp nhận yêu cầu:** Web Server (Nginx) nhận HTTP Request. Nếu cần xử lý nội dung động (ví dụ: tệp `.php`), máy chủ web đóng gói thông tin và gửi đến FastCGI Process Manager.
3. **Phân phối và xử lý:** Process Manager chọn một worker đang rảnh và chuyển giao yêu cầu. Worker thực thi mã logic, tạo ra phản hồi.
4. **Phản hồi và tái sử dụng:** Worker trả kết quả về cho Web Server, sau đó **không bị hủy** mà quay lại trạng thái chờ để xử lý request tiếp theo.

> **Ví dụ so sánh:** FastCGI giống như một nhà hàng với đội nhân viên cố định thường trực. Thay vì thuê người mới cho mỗi đơn hàng (CGI), nhà hàng có đội nhân viên sẵn sàng phục vụ liên tục, xử lý đơn hàng xong lại tiếp tục phục vụ đơn tiếp theo.

**Phương thức giao tiếp**

Web Server và FastCGI Process Manager giao tiếp qua hai cách:

- **Unix Domain Socket (UDS):** Giao tiếp qua bộ nhớ nhân của hệ điều hành (ví dụ: `/run/php/php-fpm.sock`). Tốc độ cực nhanh nhưng chỉ dùng được khi cả hai cùng máy chủ vật lý.
- **TCP/IP Socket:** Giao tiếp qua mạng (ví dụ: `127.0.0.1:9000`). Cho phép triển khai kiến trúc phân tán, Web Server và Application Server có thể đặt ở các máy chủ khác nhau - tạo nền tảng cho Load Balancing.

**Các tính năng kỹ thuật nổi bật**

- **Đa hợp (Multiplexing):** Truyền nhiều luồng yêu cầu/phản hồi qua một kết nối duy nhất.
- **Phân tách đặc quyền (Privilege Separation):** Các tiến trình FastCGI có thể chạy dưới các tài khoản hệ thống khác nhau, tăng tính bảo mật.
- **Quản lý vòng đời linh hoạt:** Tự động tăng/giảm số lượng worker theo lưu lượng, tự động khởi động lại worker gặp lỗi.

**Ưu điểm**

- Hiệu năng vượt trội so với CGI: loại bỏ overhead khởi tạo môi trường cho từng request.
- Khả năng mở rộng tốt với kiến trúc phân tán.
- Sự cố tại Application Server không làm sập Web Server.

**Thách thức**

- **Rò rỉ bộ nhớ (Memory Leak):** Vì tiến trình duy trì liên tục, nếu code ứng dụng quản lý bộ nhớ kém, RAM bị tích lũy dần. Cần cấu hình tự động khởi động lại sau một số request nhất định (ví dụ: tham số `pm.max_requests` trong PHP-FPM).
- **Cấu hình phức tạp:** Cần kiến thức chuyên sâu để tinh chỉnh số lượng worker, timeout và các tham số hiệu năng.

**Ứng dụng thực tiễn: Nginx + PHP-FPM**

Kết hợp Nginx và PHP-FPM là ví dụ kinh điển nhất của mô hình FastCGI:

- **Nginx** tiếp nhận tất cả HTTP Request, xử lý nội dung tĩnh trực tiếp, và ủy thác các request PHP cho PHP-FPM qua giao thức FastCGI (directive `fastcgi_pass`).
- **PHP-FPM** (PHP FastCGI Process Manager) duy trì một pool các tiến trình PHP, xử lý mã PHP và trả kết quả về cho Nginx.

Kiến trúc này giúp hệ thống vừa xử lý hàng chục nghìn kết nối đồng thời (nhờ Nginx), vừa thực thi logic ứng dụng hiệu quả (nhờ PHP-FPM).

#### 4.4 WSGI (Web Server Gateway Interface)

**Định nghĩa**

WSGI (Web Server Gateway Interface - được quy định trong chuẩn PEP 3333) không phải là một phần mềm hay máy chủ cụ thể, mà là một **bản đặc tả tiêu chuẩn giao tiếp (interface specification)** dành riêng cho hệ sinh thái ngôn ngữ **Python**. WSGI quy định cách thức chuẩn mực để một Web Server truyền thống (Nginx, Apache) giao tiếp với một ứng dụng web Python.

**Bối cảnh ra đời**

Trước khi có WSGI, mỗi framework Python (Django, Flask, Pyramid,...) thường bị "khóa chặt" vào một loại máy chủ web nhất định. Sự ra đời của chuẩn WSGI tạo ra một giao diện chung, cho phép bất kỳ Web Server nào hỗ trợ WSGI đều có thể chạy bất kỳ ứng dụng Python nào tuân theo chuẩn WSGI.

**Kiến trúc triển khai thực tế (3 lớp)**

Một ứng dụng Python trong môi trường sản xuất thường không đứng độc lập mà tuân theo kiến trúc 3 lớp:

```
[Internet] → Nginx/Apache → WSGI Server (Gunicorn/uWSGI) → Python App (Flask/Django)
```

- **Lớp 1 - Web Server (Nginx/Apache):** Đứng ngoài cùng, tiếp nhận HTTP Request, xử lý SSL, phục vụ nội dung tĩnh, chặn các kết nối độc hại. Không thể trực tiếp chạy mã Python.
- **Lớp 2 - WSGI Server (Gunicorn/uWSGI):** Đóng vai trò cầu nối. Nhận request từ Nginx, đóng gói thành môi trường dữ liệu chuẩn WSGI (một Python dictionary), rồi gọi ứng dụng Python để xử lý. Nhận kết quả và trả ngược về cho Nginx.
- **Lớp 3 - WSGI Application (Flask/Django):** Chứa logic nghiệp vụ của ứng dụng. Nhận dữ liệu từ WSGI Server, xử lý (truy vấn database, tính toán,...) và trả về kết quả.

> **Ví dụ thực tế:** Khi triển khai một ứng dụng Flask lên server, bạn cần:
> 1. Cài Nginx để nhận request từ Internet.
> 2. Cài Gunicorn để làm WSGI Server, kết nối Nginx với Flask.
> 3. Ứng dụng Flask xử lý logic.
>
> Nếu sau này muốn đổi sang dùng Apache thay Nginx, hoặc đổi Gunicorn sang uWSGI, bạn không cần sửa một dòng code Flask nào - vì tất cả đã tuân theo chuẩn WSGI.

**Ý nghĩa và ưu điểm**

- **Giải quyết sự phân mảnh:** Tạo ra một ngôn ngữ chung cho toàn bộ hệ sinh thái Python web.
- **Tính độc lập và linh hoạt (Decoupling):** Có thể tự do thay đổi Web Server hay WSGI Server mà không cần sửa code ứng dụng.
- **Tối ưu hóa hiệu năng và bảo mật:** Tách bạch nhiệm vụ xử lý mạng (cho Nginx) và logic ứng dụng (cho framework), giúp hệ thống ổn định và an toàn hơn. Bản thân Flask/Django có sẵn development server rất yếu, không thể chịu tải thực tế - cần Gunicorn để đảm bảo hiệu năng.

**So sánh CGI, FastCGI và WSGI**

| Tiêu chí | CGI | FastCGI | WSGI |
| :--- | :--- | :--- | :--- |
| **Phạm vi** | Đa ngôn ngữ | Đa ngôn ngữ | Chỉ Python |
| **Mô hình xử lý** | Tiến trình mới mỗi request | Tiến trình thường trú | Tiến trình thường trú |
| **Hiệu năng** | Thấp | Cao | Cao |
| **Độ phức tạp** | Thấp | Trung bình | Trung bình |
| **Ứng dụng điển hình** | Hệ thống cũ, ít dùng | PHP-FPM + Nginx | Gunicorn + Flask/Django |
